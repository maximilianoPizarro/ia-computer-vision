= CRD reference

Helm templates under `charts/all/` are the source of truth. This guide shows simplified examples with links to official product documentation.

== Validated Patterns Operator

Documentation: link:https://validatedpatterns.io/[Validated Patterns]

=== Pattern

`gitops.hybrid-cloud-patterns.io/v1alpha1` — the single custom resource that bootstraps the entire pattern. The Validated Patterns Operator watches this object, installs OpenShift GitOps, and points it at this repository's `clustergroup` chart with the values files listed in `extraValueFiles`.

[IMPORTANT]
====
See link:../pattern-cr-guide/[Pattern CR guide] for the full decision table (Scenarios A–E) and copy-paste ready files in `examples/pattern-cr/`. The examples below are annotated references — start with the guide to pick the right scenario.
====

==== Hub-only CPU on a fresh multi-node cluster (default)

This is the **most common** install path: one hub cluster, CPU inference, no pre-installed operators, no east or west spokes.

[source,yaml]
----
apiVersion: gitops.hybrid-cloud-patterns.io/v1alpha1
kind: Pattern
metadata:
  name: ia-computer-vision
  namespace: openshift-operators
spec:
  clusterGroupName: hub
  extraValueFiles:
    - /values-hub-only.yaml         # <1>
  gitSpec:
    targetRepo: https://github.com/maximilianoPizarro/ia-computer-vision.git
    targetRevision: main
  multiSourceConfig:
    enabled: true
    clusterGroupChartVersion: "0.9.*"
    helmRepoUrl: https://charts.validatedpatterns.io
----
<1> Disables Skupper/ACM spoke import; runs NeuroFace, Kafka, and CPU inference locally on the hub. File: `examples/pattern-cr/hub-only-cpu.yaml`.

==== Hub + east + west spokes (CPU)

Three-cluster topology. Install spokes first, then the hub with inline spoke tokens:

[source,yaml]
----
apiVersion: gitops.hybrid-cloud-patterns.io/v1alpha1
kind: Pattern
metadata:
  name: ia-computer-vision
  namespace: openshift-operators
spec:
  clusterGroupName: hub
  gitSpec:
    targetRepo: https://github.com/maximilianoPizarro/ia-computer-vision.git
    targetRevision: main
  multiSourceConfig:
    enabled: true
    clusterGroupChartVersion: "0.9.*"
    helmRepoUrl: https://charts.validatedpatterns.io
  extraParameters:
    - name: spokeCredentials.mode
      value: inline
    - name: spokeCredentials.clusters.east.token
      value: "<EAST_SA_TOKEN>"
    - name: spokeCredentials.clusters.east.apiUrl
      value: "https://api.<EAST_DOMAIN>:6443"
    - name: spokeCredentials.clusters.west.token
      value: "<WEST_SA_TOKEN>"
    - name: spokeCredentials.clusters.west.apiUrl
      value: "https://api.<WEST_DOMAIN>:6443"
----

Do **not** add `values-hub-only.yaml` in this scenario. File: `examples/pattern-cr/hub-spoke-cpu.yaml`.

==== Hub-only with GPU, on a cluster where operators are already installed for you

This is the live-verified `Pattern` CR for a hub-only install with GPU-backed model serving on a cluster that already has Node Feature Discovery, the NVIDIA GPU Operator, OpenShift Service Mesh, OpenShift AI, and Connectivity Link pre-installed, and only a single node available:

[source,yaml]
----
apiVersion: gitops.hybrid-cloud-patterns.io/v1alpha1
kind: Pattern
metadata:
  name: ia-computer-vision
  namespace: openshift-operators
spec:
  clusterGroupName: hub
  extraValueFiles:
    - /values-hub-gpu.yaml          # <1>
    - /values-hub-single-node.yaml  # <2>
    - /values-hub-rhpds.yaml        # <3>
    - /values-hub-only.yaml         # <4>
  gitSpec:
    targetRepo: https://github.com/maximilianoPizarro/ia-computer-vision.git
    targetRevision: main
  multiSourceConfig:
    enabled: true
    clusterGroupChartVersion: "0.9.*"
    helmRepoUrl: https://charts.validatedpatterns.io
----
<1> GPU-backed vLLM model serving on the hub.
<2> Scales workshop user counts down to fit a single node's pod-count ceiling.
<3> Skips duplicate `OperatorGroup`s, pins the ZTunnel version to the pre-installed Service Mesh, and points `models-as-a-service` at the pre-installed Connectivity Link namespace. Omit this file entirely on a from-scratch, multi-node hub where the pattern should install its own copies of these operators.
<4> Hub-only topology — no east/west spoke clusters.

File: `examples/pattern-cr/hub-only-gpu-single-node-preinstalled.yaml`.

[NOTE]
====
Create this object with `oc apply -f pattern-cr.yaml`, or through the console: *Operators* -> *Installed Operators* -> *Validated Patterns Operator* -> *Pattern* tab -> *Create Pattern* -> *YAML view*. It replaces running `./pattern.sh make install` with `TARGET_CLUSTERGROUP=hub` plus manually editing `values-global.yaml`'s `extraValueFiles` — both paths converge on the same `Pattern` object.
====

See link:../pattern-cr-guide/[Pattern CR guide] for the decision table, link:../ideas-for-customization/#_configuring_gpu_inference_on_a_cluster_with_pre_installed_operators[Configuring GPU inference on a cluster with pre-installed operators] for the pre-installed-operator walkthrough, and link:../cluster-sizing/[Cluster sizing] for single-node `maxPods` guidance.

*Connection*: Pattern → `patterns-operator` → OpenShift GitOps (`vp-gitops` namespace) → `ia-computer-vision-hub` umbrella Application → every Application listed in `clusterGroup.applications` across `values-hub.yaml` and the layered `extraValueFiles`.

== How a self-service OIDC client becomes a rate-limit tier

When a workshop user runs the OIDC credentials self-service scaffolder template in Developer Hub, the chosen plan tier flows through the gateway stack in five steps:

. The user selects `planTier` (`free` or `gold`) on the template form.
. The template creates a Keycloak client with an `oidc-hardcoded-claim-mapper` protocol mapper that embeds `plan: <tier>` in every JWT.
. The client obtains a token from realm `cv` (`client_credentials` or Bearer flow).
. Kuadrant `AuthPolicy` (`authpolicy-cv` on `neuroface-cv-lb`) validates the JWT issuer (`keycloak.apps.<hub-domain>/realms/cv`) and admits the request.
. `RateLimitPolicy` and `PlanPolicy` on `neuroface-cv-lb` match `auth.identity.plan == "free"` or `"gold"` and apply `/v1/predict` limits (100/h or 500/h).

The MaaS route (`ai-maas`) uses the same dual-auth pattern (`APIKEY` or OIDC JWT) with `PlanPolicy` and `TokenRateLimitPolicy` tiers on `/v1/chat/completions`.

=== OIDCPolicy

`extensions.kuadrant.io/v1alpha1` — browser redirect OIDC login on Gateway API HTTPRoutes (interactive sessions).

[NOTE]
====
`neuroface-cv-lb` deliberately does not use `OIDCPolicy`. That route serves API clients with Bearer tokens, not browser logins. An `OIDCPolicy` redirect breaks CORS preflight (OPTIONS requests cannot carry a token). Realm `cv` on `neuroface-cv-lb` uses `AuthPolicy` + `jwt-users` instead (see below).
====

[source,yaml]
----
apiVersion: extensions.kuadrant.io/v1alpha1
kind: OIDCPolicy
metadata:
  name: oidc-neuroface
  namespace: neuroface-gateway-system
spec:
  provider:
    issuerURL: https://sso.apps.<hub-domain>/realms/neuroface
    clientID: client-neuroface-user1
    authorizationEndpoint: https://sso.apps.<hub-domain>/realms/neuroface/protocol/openid-connect/auth
    tokenEndpoint: https://sso.apps.<hub-domain>/realms/neuroface/protocol/openid-connect/token
    redirectURI: https://neuroface.apps.<hub-domain>/auth/callback
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: neuroface-app-lb
  auth:
    tokenSource:
      authorizationHeader:
        prefix: Bearer
        name: Authorization
      cookie:
        name: jwt
----

*Connection*: OIDCPolicy → HTTPRoute → Gateway → backend Service (Skupper listeners on hub reach spoke connectors). Per-user scaffolded NeuroFace instances on spokes use the same redirect pattern against their local RHBK realm.

Chart: `charts/all/neuroface-gateway/templates/oidcpolicy-realms.yaml`

=== AuthPolicy

`kuadrant.io/v1` — API key and/or JWT authentication. A single `AuthPolicy` can declare multiple
named identity sources under `rules.authentication`; Authorino evaluates all of them and admits
the request if *any one* succeeds. The AI Gateway route (`ai-maas`) uses this to accept either a
Kuadrant API key or a Keycloak realm-`maas` OIDC token on the same endpoint ("dual auth").

[source,yaml]
----
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: ai-maas-auth
  namespace: ai-gateway-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-maas
  when:
    - predicate: request.method != "OPTIONS"
  rules:
    authentication:
      api-key-users:
        apiKey:
          allNamespaces: true
          selector:
            matchLabels:
              app: workshop-llm-tokens
              kuadrant.io/apikey: "true"
        credentials:
          authorizationHeader:
            prefix: APIKEY
      jwt-users:
        jwt:
          issuerUrl: https://keycloak.apps.<hub-domain>/realms/maas
----

*Connection*: AuthPolicy → HTTPRoute → APIProduct (Developer Hub catalog) → API key Secret, or → Keycloak realm (OIDC JWT).

Chart: `charts/all/workshop-kuadrant-apis/templates/policies.yaml`

=== RateLimitPolicy

`kuadrant.io/v1` — per-route request limits keyed off authenticated identity (JWT claims or client IP).

[source,yaml]
----
apiVersion: kuadrant.io/v1
kind: RateLimitPolicy
metadata:
  name: neuroface-cv-ratelimit
  namespace: neuroface-gateway-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: neuroface-cv-lb
  limits:
    predict-free:
      when:
        - predicate: 'request.path.matches("^/v1/predict")'
        - predicate: 'has(auth.identity) && has(auth.identity.plan) && auth.identity.plan == "free"'
      counters:
        - expression: auth.identity.sub
      rates:
        - limit: 100
          window: 1h
    predict-gold:
      when:
        - predicate: 'request.path.matches("^/v1/predict")'
        - predicate: 'has(auth.identity) && has(auth.identity.plan) && auth.identity.plan == "gold"'
      counters:
        - expression: auth.identity.sub
      rates:
        - limit: 500
          window: 1h
    predict-default:
      when:
        - predicate: 'request.path.matches("^/v1/predict")'
        - predicate: '!(has(auth.identity) && has(auth.identity.plan))'
      counters:
        - expression: request.headers['x-forwarded-for']
      rates:
        - limit: 30
          window: 1m
----

*Connection*: RateLimitPolicy → HTTPRoute. The `plan` JWT claim (from the self-service template `planTier` parameter) selects the `predict-free` or `predict-gold` tier.

Chart: `charts/all/neuroface-gateway/templates/ratelimit-policy.yaml`

=== PlanPolicy

`extensions.kuadrant.io/v1alpha1` — publish discoverable rate-limit tiers in the Kuadrant DevPortal plugin (Developer Hub API entity Kuadrant tab).

[source,yaml]
----
apiVersion: extensions.kuadrant.io/v1alpha1
kind: PlanPolicy
metadata:
  name: neuroface-cv-plans
  namespace: neuroface-gateway-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: neuroface-cv-lb
  plans:
    - tier: free
      predicate: |
        has(auth.identity) && has(auth.identity.plan) && auth.identity.plan == "free"
      limits:
        custom:
          - limit: 100
            window: 1h
    - tier: gold
      predicate: |
        has(auth.identity) && has(auth.identity.plan) && auth.identity.plan == "gold"
      limits:
        custom:
          - limit: 500
            window: 1h
----

*Connection*: PlanPolicy → HTTPRoute → DevPortal discovered-plans UI. Pairs with RateLimitPolicy on the same route.

Chart: `charts/all/neuroface-gateway/templates/plan-policy.yaml`

=== TokenRateLimitPolicy

`kuadrant.io/v1alpha1` — token-budget rate limits for LLM endpoints (counts tokens, not requests).

[source,yaml]
----
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: ai-maas-token-limits
  namespace: ai-gateway-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-maas
  limits:
    free:
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"
        - predicate: |
            auth.identity.metadata.annotations["secret.kuadrant.io/plan-id"] == "free" ||
            auth.identity.groups.split(",").exists(g, g == "free")
      counters:
        - expression: auth.identity.userid
    oidc:
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"
        - predicate: has(auth.identity.iss)
      counters:
        - expression: auth.identity.sub
----

*Connection*: TokenRateLimitPolicy → HTTPRoute (`ai-maas`). Works with dual-auth AuthPolicy and PlanPolicy tiers (`free`, `gold`, `oidc`).

Chart: `charts/all/workshop-kuadrant-apis/templates/policies.yaml`

[IMPORTANT]

== Red Hat Advanced Cluster Management (ACM)

Documentation: link:https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes[Red Hat Advanced Cluster Management]

The `acm-hub-spoke` chart automates import by creating these same three objects from `spoke-credentials` stored in Vault (see link:../getting-started/#importing-spoke-clusters-into-rhACM[Importing spoke clusters into RHACM]). Use the YAML below to import a spoke manually — for example, from the RHACM console's *Import cluster* wizard, or when you do not want to store a spoke token in Vault.

=== ManagedCluster

`cluster.open-cluster-management.io/v1` — registers a spoke cluster with the hub. Apply this **first**; RHACM creates a namespace named after the cluster (`spec.hubAcceptsClient: true` accepts the join request; do not pre-create this namespace yourself, or OpenShift terminates it before RHACM claims it).

[source,yaml]
----
apiVersion: cluster.open-cluster-management.io/v1
kind: ManagedCluster
metadata:
  name: east                              # <1>
  labels:
    name: east
    region: east
    clusterGroup: east                    # <2>
    cloud: auto-detect
    vendor: OpenShift
    cluster.open-cluster-management.io/clusterset: global
spec:
  hubAcceptsClient: true
  leaseDurationSeconds: 60
----
<1> Replace with `west` for the west spoke.
<2> `clusterGroup` must match `east`/`west` — this pattern's Applications and dashboards select on it.

=== Secret (`auto-import-secret`)

`v1/Secret` — a well-known name and shape that RHACM's import controller watches for in the newly created cluster namespace. Wait for `oc get namespace east` to succeed before applying this (RHACM creates the namespace asynchronously after the `ManagedCluster` above).

[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: auto-import-secret
  namespace: east                          # <1>
type: Opaque
stringData:
  autoImportRetry: "5"
  server: https://api.<east_domain>:6443   # <2>
  token: <east_cluster_admin_token>         # <3>
----
<1> Must match the `ManagedCluster` name/namespace from the previous step.
<2> The spoke's own API server URL (`oc whoami --show-server` on the spoke).
<3> A long-lived cluster-admin token from the spoke. This pattern auto-creates an `acm-import` ServiceAccount with `cluster-admin` (via the `platform-users` chart) on every cluster — generate one with `oc create token -n kube-system acm-import --duration=87600h`.

[WARNING]
====
Never commit this token to Git. Apply it directly with `oc apply -f -` or `oc create secret generic auto-import-secret -n east --from-literal=...`, and delete the manifest file afterward if you wrote it to disk.
====

=== KlusterletAddonConfig

`agent.open-cluster-management.io/v1` — enables the RHACM add-ons (policy controller, search, application manager) on the spoke once import succeeds. Apply this after the `ManagedCluster`; it can be applied in the same batch as the `Secret` above since RHACM processes it once the klusterlet agent is running.

[source,yaml]
----
apiVersion: agent.open-cluster-management.io/v1
kind: KlusterletAddonConfig
metadata:
  name: east
  namespace: east
spec:
  clusterName: east
  clusterNamespace: east
  clusterLabels:
    cloud: auto-detect
    vendor: OpenShift
  applicationManager:
    enabled: true
  certPolicyController:
    enabled: true
  iamPolicyController:
    enabled: true
  policyController:
    enabled: true
  searchCollector:
    enabled: true
----

*Connection*: `ManagedCluster` → RHACM creates the `east` namespace → `Secret/auto-import-secret` (in that namespace) → RHACM's import controller installs the klusterlet agent on the spoke using the token → `KlusterletAddonConfig` → RHACM add-ons activate. Apply all three (`ManagedCluster` first, then the other two once the namespace exists) with `oc apply -f -` on the **hub**, not the spoke.

Verify the import succeeded:

[source,bash]
----
$ oc get managedcluster east -o jsonpath='{.status.conditions[?(@.type=="ManagedClusterConditionAvailable")].status}{"\n"}'
$ oc get pods -n open-cluster-management-agent --context east   # run on the spoke
----

.Expected output
[source,text]
----
True
----

Chart: `charts/all/acm-hub-spoke/templates/managed-clusters.yaml` (the automated equivalent, driven by Vault-stored `spoke-credentials`).

=== ZTunnel

`sailoperator.io/v1` — deploys the ambient mesh data plane (`ztunnel` DaemonSet). Required for L4 mTLS and Istio metrics (`istio_requests_total`, `istio_tcp_*`) consumed by Kiali and Grafana.

[source,yaml]

apiVersion: sailoperator.io/v1

metadata:
  name: default
  namespace: istio-system
spec:
  namespace: ztunnel
  version: v1.28.8
----

*Connection*: ZTunnel → `ztunnel` namespace DaemonSet on every node. Without this CR, `oc get istio default -n istio-system` reports `ZTunnelNotFound` and mesh telemetry panels show no data.
