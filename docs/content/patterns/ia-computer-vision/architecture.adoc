= AI Computer Vision architecture

This document describes the hub-spoke topology, component placement, traffic flow, security model, connectivity, and GitOps workflow for the AI Computer Vision pattern. It also explains why specific Red Hat products were chosen over alternatives.

== Overview

The AI Computer Vision pattern uses a **hub-spoke** model with one central control plane cluster and two symmetric edge inference clusters. This architecture separates fleet management and developer platform services from latency-sensitive inference workloads.

You deploy the pattern on three independent OCP 4.20 or later clusters:

* **Hub** — fleet control, secrets, centralized security, developer platform, and external ingress gateway
* **East spoke** — edge inference, observability, and service mesh
* **West spoke** — identical edge stack for geographic redundancy and load distribution

=== Why hub-spoke

Computer vision inference runs close to cameras and sensors for low-latency response. Centralizing policy, secrets, and observability on the hub means edge clusters remain lightweight and recoverable. If a spoke goes offline, the hub retains the full audit trail and can redeploy the spoke from Git state alone. RHACM provides the fleet management layer, and OpenShift GitOps provides the reconciliation engine.

== Spoke cluster components

East and west spokes run identical software stacks. Helm overrides set `clusterName` to `east` or `west` and `clusterSuffix` for observability labeling.

[cols="2,3,4",options="header"]
|===
| Component | Role | Why this product

| NeuroFace
| Edge AI application with Kafka event streaming for computer vision scenarios
| Reference workload demonstrating multi-cluster inference. Receives images, runs detection, streams events.

| OpenVINO Model Server (OVMS) / YOLO CV
| PPE (Personal Protective Equipment) detection inference in `neuroface-cv` namespace
| CPU-efficient model serving using Intel OpenVINO runtime. Suitable for edge nodes without GPU. YOLO provides real-time object detection.

| Skupper 2.x (Service Interconnect)
| Application-layer connectivity between spokes and hub without VPN or network peering
| Skupper creates encrypted TLS links at the application layer, tolerating NAT, firewalls, and heterogeneous network topologies common at edge sites. The alternative (VPN or VPC peering) requires network team involvement and exposes entire cluster networks rather than individual services.

| Red Hat OpenShift Service Mesh 3.2 (ambient mode)
| Mutual TLS (mTLS), L4 telemetry, and authorization policies without sidecar injection
| Ambient mode eliminates the resource and operational overhead of sidecar proxies on edge nodes where CPU and memory are constrained. Each node runs a ztunnel DaemonSet that intercepts traffic transparently. The alternative (sidecar mode) doubles container count per pod and complicates debugging.

| RHACS Secured Cluster
| Registers spoke with hub ACS Central; enforces runtime security policies locally
| Consistent vulnerability scanning and policy enforcement at the edge, correlated with the hub Central dashboard.

| Observability stack (Grafana, OpenTelemetry, Kiali)
| Local metrics collection and federation to hub Thanos; distributed tracing
| Each spoke exports metrics via Thanos sidecar. OpenTelemetry collectors emit traces that appear in the hub Grafana and Kiali dashboards.

| OpenShift DevSpaces
| Cloud IDE workspaces with per-user GitLab credentials
| Remote development for edge AI application teams without local environment setup.

| platform-users
| HTPasswd workshop users and namespace view RBAC on spokes
| Consistent lab credentials across hub and edge clusters for workshop and demo events.
|===

=== Per-user NeuroFace scaffolding (workshop)

Developer Hub software template `ai-computer-vision` scaffolds a GitLab repo in the shared `ws-workshop` group. An Argo CD ApplicationSet on the **hub** (`vp-gitops`) watches for repos matching `neuroface-*` and deploys manifests from the `k8s/` directory to the spoke named in `k8s/target-cluster.yaml`. See link:../scaffolding-and-secrets/[Scaffolding and secrets] for the full operator flow.

Each user namespace (`neuroface-userN`) contains:

* NeuroFace frontend and backend (shared images v1.5.1; Tekton builds to internal registry)
* Per-user RHBK with biometric SPI (facial enrollment + 2FA login)
* HTTPRoute on the shared spoke Gateway with path `/user/userN/`
* RHCL `AuthPolicy` with OIDC issuer pointing to the user's RHBK realm

Workshop credentials remain consistent: `userN` / `Welcome123!` (same as HTPasswd). First RHBK login triggers biometric enrollment via webcam.

== Traffic flow: inference request end-to-end

This section describes each hop of an inference request from an external client through the hub gateway to edge model serving and back.

.Inference request traffic flow
image::inference-traffic-flow.png[Sequence diagram showing inference request flow through RHCL gateway to NeuroFace and OVMS,align=center,width=100%]

=== Step-by-step flow

[cols="1,2,5",options="header"]
|===
| Step | Component | What happens

| 1
| External client
| Sends an HTTPS POST request to the RHCL Gateway HTTPRoute host (for example, `https://neuroface.apps.<hub_domain>/api/detect`). The request contains an image payload for PPE detection.

| 2
| RHCL Gateway (hub)
| The Kuadrant-managed Gateway evaluates HTTPRoute rules. If `AuthPolicy` is configured, the gateway validates API keys or JWT tokens. If `RateLimitPolicy` is configured, Limitador enforces per-client rate limits. The request passes to the backend selector.

| 3
| HTTPRoute backend selection
| The HTTPRoute defines two `backendRefs` with weight 50 each — one pointing to the east spoke NeuroFace service and one to the west spoke service. The gateway selects one backend based on the weighted round-robin algorithm.

| 4
| Skupper VAN tunnel
| The selected backend reference resolves to a Skupper listener on the hub. Skupper tunnels the request over a TLS-encrypted AMQP link to the corresponding spoke `service-interconnect` namespace, where a connector forwards it to the NeuroFace `ClusterIP` service.

| 5
| Service Mesh ambient (spoke)
| On the spoke, the ztunnel DaemonSet intercepts the request at the node level and applies mTLS transparently. No sidecar proxy is involved. L4 metrics (connection count, bytes transferred, latency) are emitted to OpenTelemetry collectors.

| 6
| NeuroFace application (spoke)
| The NeuroFace pod receives the image, applies preprocessing, and forwards it to the OVMS inference endpoint in the `neuroface-cv` namespace via an internal `ClusterIP` service call.

| 7
| OVMS / YOLO CV (spoke)
| OVMS loads the YOLO model from the configured model store (PVC or S3) and runs object detection. It returns a JSON response with detection bounding boxes, classes, and confidence scores.

| 8
| Response path
| The detection JSON returns through NeuroFace -> Skupper tunnel -> RHCL Gateway -> client. OpenTelemetry trace context propagates across all hops, allowing end-to-end latency analysis in Grafana and Kiali.
|===

=== Protocols used

* **Client -> RHCL Gateway**: HTTPS (TLS terminated at OpenShift Route/Gateway)
* **RHCL Gateway -> Skupper**: Internal cluster networking + Skupper AMQP over TLS
* **Skupper -> spoke service**: TLS-encrypted application-layer tunnel (no VPN)
* **Intra-spoke (NeuroFace -> OVMS)**: HTTP/gRPC over mTLS (Service Mesh ambient ztunnel)
* **Metrics/traces**: OpenTelemetry Protocol (OTLP) over gRPC to collectors

== Design decisions

This section explains why specific products and architectures were chosen. These are the questions a Validated Patterns architecture reviewer asks during informal review.

=== Why RHCL for the gateway instead of native OCP Routes?

OCP Routes use HAProxy for ingress and support only basic round-robin across multiple backends. RHCL provides:

* **Gateway API HTTPRoute** with weighted backend references (50/50 split by default, configurable per-backend)
* **Kuadrant policy integration**: `AuthPolicy` for API key and JWT validation, `RateLimitPolicy` for per-client throttling, `DNSPolicy` for multi-cluster DNS
* **Standards-based**: Gateway API is the Kubernetes standard for ingress, replacing Ingress resources

An OCP Route cannot split traffic between two external backends by weight; it always round-robins to endpoints within a single service. RHCL HTTPRoute treats east and west as independent backends with configurable weights.

=== Why Skupper instead of VPN or network peering?

Skupper operates at the application layer (Layer 7), not the network layer:

* **No VPN configuration**: Skupper does not require IPsec tunnels, VPC peering, or Transit Gateway setup. It initiates outbound TLS connections from each spoke, which works through NAT and firewalls without opening inbound ports.
* **Selective service exposure**: Only the services you configure as listeners and connectors are reachable across clusters. With a VPN, entire cluster networks are exposed.
* **Portable across providers**: The same Skupper configuration works on AWS, Azure, GCP, bare metal, and disconnected environments. VPN configurations are provider-specific.

=== Why Service Mesh ambient mode instead of sidecar mode?

Red Hat OpenShift Service Mesh 3.x supports both sidecar and ambient mode. This pattern uses ambient mode because:

* **Lower resource overhead**: Ambient mode runs a single ztunnel DaemonSet per node instead of injecting an Envoy sidecar into every pod. On edge nodes with limited CPU and memory, this reduces the per-pod resource footprint.
* **No application restart required**: Enabling mesh for a namespace does not require pod restarts. The ztunnel transparently intercepts traffic at the node level.
* **L4 mTLS by default**: All pod-to-pod traffic within mesh-enrolled namespaces gets mTLS encryption without application changes.
* **Trade-off**: Ambient mode provides L4 (transport) features only. L7 features (HTTP routing, retries, header manipulation) require waypoint proxies, which this pattern does not deploy. For this inference workload, L4 mTLS and telemetry are sufficient.

== Security architecture

The pattern implements defense in depth across four layers.

=== Identity and access

* **RHBK (Keycloak)** provides OIDC SSO for Developer Hub. The `backstage` realm contains workshop users and the `developer-hub` OIDC client.
* **platform-users** provisions `user1` through `user30` with HTPasswd accounts on all clusters. Each user gets `cluster-reader` ClusterRoleBinding and `view` RoleBindings in pattern namespaces.
* **RHACM RBAC** restricts fleet operations (cluster import, policy enforcement) to cluster administrators.

=== Secrets management

image::scaffolding-vault-eso-flow.png[Vault and ESO flow — from values-secret to OpenShift Secret,align="center",width=100%]

* **Vault** stores auto-generated credentials (GitLab root password, Keycloak admin password, Developer Hub session secret) and user-provided secrets (GitLab PAT, MaaS API key).
* **ESO** projects Vault secrets into Kubernetes `Secret` objects consumed by operators and applications. The mapping is defined in `ExternalSecret` CRs within each chart.
* **No secrets in Git**: Only `ExternalSecret` references are committed. Actual secret values live in Vault and are synced at deploy time.

For operator interaction diagrams, scaffolder module descriptions, and expected results, see link:../scaffolding-and-secrets/[Scaffolding and secrets].

=== Runtime security

* **RHACS Central** on the hub collects policy violations and CVE data from **Secured Cluster** sensors on spokes.
* **ACS init bundle sync** chart distributes `SecuredCluster` registration material (certificates, endpoint) to spokes via GitOps, eliminating manual bundle exchange.

=== Network security

* **Service Mesh ambient mode** enforces mTLS between all workloads in mesh-enrolled namespaces (`istio-system`, `istio-cni`, `ztunnel` DaemonSet per node).
* **Skupper** encrypts all cross-cluster application traffic over TLS without exposing cluster SDN networks.
* **RHCL/Kuadrant** applies authentication and rate-limiting policies at the gateway layer before requests reach backend services.

== Connectivity: Skupper Service Interconnect

Skupper creates a **virtual application network (VAN)** that links services across clusters at the application layer.

On each spoke, the `spoke-interconnect` chart deploys:

* A **Site** resource that identifies the cluster as a Skupper participant
* **Listener** resources that expose remote services locally (for example, exposing hub services on the spoke)
* **Connector** resources that expose local services to the VAN (for example, making spoke NeuroFace reachable from the hub)

Skupper enables:

* Hub RHCL Gateway to reach NeuroFace and OVMS endpoints on spokes without VPN
* Cross-spoke service visibility for inference backends
* Encrypted links that tolerate NAT and firewall boundaries at edge sites

Verify Skupper resources:

[source,bash]
----
$ oc get site -n service-interconnect
$ oc get listener,connector -n service-interconnect
----

== Connectivity: RHACM managed clusters

RHACM imports east and west clusters when you apply the `clusterGroup=east` and `clusterGroup=west` labels. The `managedClusterGroups` section in `values-hub.yaml` declares these groups:

[source,yaml]
----
managedClusterGroups:
  east:
    name: east
    acmlabels:
      - name: clusterGroup
        value: east
  west:
    name: west
    acmlabels:
      - name: clusterGroup
        value: west
----

RHACM uses these labels for policy propagation and to populate managed cluster metadata that the Validated Patterns Operator injects into Helm values (for example, spoke `apps.<domain>` used by Skupper and RHCL).

== GitOps flow

The Validated Patterns Operator installs OpenShift GitOps and creates a Pattern custom resource that points to this Git repository.

. You commit changes to `values-hub.yaml`, `values-east.yaml`, or `values-west.yaml`.
. OpenShift GitOps on each cluster watches the `main` branch (configurable in the Pattern CR).
. The `clustergroup` chart (version `0.9.*` from `charts.validatedpatterns.io`) renders Argo CD Applications according to sync waves.
. Sync waves ensure operators install before their operands: wave 0 (platform base) through wave 10 (console links).
. On the hub, RHACM observes managed cluster labels and propagates policies.
. Spoke clusters reconcile their own values files independently, ensuring edge workloads remain available even if hub connectivity is temporarily interrupted.

| 3–4

| OpenShift AI, Showroom, NeuroFace gateway, NeuroFace, OVMS, DevSpaces

| Console links
|===
