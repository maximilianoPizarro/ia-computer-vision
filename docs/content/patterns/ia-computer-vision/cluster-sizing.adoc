= Cluster sizing and version matrix

This page documents cluster sizing requirements, network prerequisites, and the tested product version matrix for the AI Computer Vision pattern.

== Recommended AWS sizing

[cols="2,2,2,2",options="header"]
|===
| Cluster role | Control plane | Workers | Notes

| Hub
| 3 x `m6a.2xlarge`
| 3 x `m6a.4xlarge`
| Hub runs 15+ operators and ~450 pods. `m6a.2xlarge` workers (8 vCPU) reach 98-99% CPU request allocation. Use `m6a.4xlarge` (16 vCPU, 64 GiB) for production. For sandbox/demo, `m6a.2xlarge` works if you remove the `NoSchedule` taint from master nodes.

| East spoke
| 3 x `m6a.2xlarge`
| 3 x `m6a.2xlarge`
| Spoke workloads are lighter (~150 pods). `m6a.2xlarge` is sufficient for CPU-only inference.

| West spoke
| 3 x `m6a.2xlarge`
| 3 x `m6a.2xlarge`
| Same as east.
|===

Each `m6a.2xlarge` instance provides 8 vCPU and 32 GiB memory. Each `m6a.4xlarge` provides 16 vCPU and 64 GiB memory.

=== Single-node RHPDS sandboxes (e.g. g6.12xlarge GPU demo)

RHPDS "Red Hat OpenShift AI 3" catalog sandboxes provision a **single node** that is simultaneously control plane and worker. That node runs the full OpenShift control plane, RHACM/multicluster-engine, OpenShift Pipelines, RHOAI, and the GPU Operator *before* the pattern installs anything -- this baseline alone consumes roughly half of OpenShift's **default 250 max-pods-per-node** kubelet limit, even though the underlying instance (e.g. `g6.12xlarge`: 48 vCPU, 192 GiB RAM, 4x NVIDIA L4) has abundant CPU/memory headroom. The constraint is the pod-count ceiling, not hardware.

Symptoms of hitting this ceiling: ArgoCD's own `application-controller` pod stuck `Pending` ("Too many pods"), RHACM's `ocm-proxyserver` stuck `Pending` (breaks `clusterview`/`proxy.open-cluster-management.io` API aggregation and blocks namespace deletion cluster-wide), GitLab/DevSpaces/Developer Hub pods `Pending`, GPU model `InferenceService`s stuck `Pending`.

**Fix: raise `maxPods` via `KubeletConfig` before installing the pattern.** The default OVN-Kubernetes `hostPrefix: 23` (512 addresses per node) comfortably supports raising the limit to 500:

[source,yaml]
----
apiVersion: machineconfiguration.openshift.io/v1
kind: KubeletConfig
metadata:
  name: max-pods-single-node
spec:
  machineConfigPoolSelector:
    matchLabels:
      pools.operator.machineconfiguration.openshift.io/master: ""
  kubeletConfig:
    maxPods: 500
    podsPerCore: 0
----

Apply with `oc apply -f` and wait for the `master` `MachineConfigPool` to roll out (`oc get mcp master -w`) -- this **reboots the single node** (5-15 minutes of full cluster unavailability, including the API server). Verify with `oc get node -o jsonpath='{.items[0].status.allocatable.pods}'` (expect `500`).

With `maxPods` raised, the full hub profile (`values-hub-gpu.yaml` + `values-hub-single-node.yaml` + `values-hub-rhpds.yaml` + `values-hub-only.yaml`) fits, including GitLab, Developer Hub, DevSpaces, and RHBK -- `values-hub-gpu-minimal.yaml` (which disables all of those) is only needed as a fallback on sandboxes where `maxPods` cannot be changed.

=== Sandbox sizing (m6a.2xlarge workers)

In a tested sandbox deployment with `m6a.2xlarge` workers (3 control + 3 worker nodes per cluster), the hub reached **98-99% CPU request allocation** on the 3 schedulable worker nodes with all operators running. The following adjustments were needed:

* Scale `rhods-dashboard` to 1 replica (default 2, each pod uses 3.1 CPU requests across 9 containers)
* Reduce ACS Central scanner-v4 resource requests
* Remove `NoSchedule` taint from master nodes to allow scheduling on all 6 nodes (taints reappear after reconciliation; use a MachineConfig or `Scheduler` CR with `mastersSchedulable: true` to persist)

[IMPORTANT]
====
For production deployments, use `m6a.4xlarge` or larger workers on the hub. The hub runs RHACM, ACS Central (scanner, scanner-v4), GitLab, Developer Hub, OpenShift AI, Keycloak, and Vault — each operator contributes significant CPU requests even when idle.
====

== Minimum requirements

* OCP 4.20 or later on all clusters
* Three worker nodes per cluster (minimum); hub recommended `m6a.4xlarge` or equivalent
* Sufficient persistent storage for Vault, GitLab, and OpenShift AI on the hub
* Default StorageClass configured and provisioning PVCs
* Hub total CPU capacity: minimum 32 vCPU allocatable across workers (with operator defaults)

== Network requirements

Inter-cluster communication is required for RHACM, Skupper, and RHACS. The following table lists the ports and protocols used:

[cols="3,1,1,3",options="header"]
|===
| Purpose | Port | Protocol | Direction

| RHACM klusterlet registration
| 6443
| HTTPS
| Spoke -> Hub API server

| RHACM agent communication
| 443
| HTTPS
| Spoke -> Hub `multicloud-console` route

| Skupper inter-router link
| 8443
| TLS/AMQP
| Spoke -> Hub (outbound from spoke; NAT-friendly)

| Skupper data channel
| 45671
| TLS/AMQP
| Bidirectional between linked sites

| RHACS sensor to Central
| 8443
| HTTPS/gRPC
| Spoke -> Hub `central-stackrox` route

| Thanos sidecar federation
| 10901
| gRPC
| Hub -> Spoke Thanos sidecar (via Service route)

| OpenTelemetry collector
| 4317
| gRPC (OTLP)
| Within each cluster (collector -> Thanos/Grafana)
|===

[IMPORTANT]
====
Skupper initiates all connections outbound from spokes. You do not need to open inbound ports on spoke clusters for Skupper connectivity. However, the hub must accept inbound connections from spokes on port 8443 via the Skupper inter-router route.
====

Recommended maximum inter-cluster latency:

* **Skupper**: 200 ms RTT for responsive inference traffic forwarding
* **RHACM**: 500 ms RTT (control plane only, tolerant of latency)
* **Thanos federation**: 100 ms RTT for responsive Grafana dashboards
* **RHACS**: 500 ms RTT (sensor heartbeat is tolerant)

== GPU acceleration (optional)

For GPU-accelerated inference on spokes, add worker nodes with NVIDIA GPUs:

* AWS `g4dn.xlarge` (NVIDIA T4 — 16 GiB GPU memory, suitable for YOLO inference)
* AWS `g5.2xlarge` (NVIDIA A10G — 24 GiB GPU memory, suitable for larger models)

Install these operators on spoke clusters before enabling GPU workloads:

. **Node Feature Discovery (NFD) Operator** — detects GPU hardware on nodes
. **NVIDIA GPU Operator** — installs drivers and device plugin

See link:../ideas-for-customization/#_enabling_gpu_workers_on_spokes[Enabling GPU workers on spokes] for step-by-step instructions.

For GPU-backed model serving on the **hub**, including on a cluster where NFD, the GPU Operator, Service Mesh, or OpenShift AI are already pre-installed for you, see link:../ideas-for-customization/#_enabling_gpu_workers_on_the_hub_real_local_model_download_and_serving[Enabling GPU workers on the hub] and link:../ideas-for-customization/#_configuring_gpu_inference_on_a_cluster_with_pre_installed_operators[Configuring GPU inference on a cluster with pre-installed operators].

=== Per-user NeuroFace scaffolding (30 users)

Each scaffolded user deploys NeuroFace + RHBK biometric on a spoke (~650m CPU, 896Mi memory per user). With 15 users per spoke, plan for ~10 vCPU and ~13 GiB additional capacity beyond the base spoke stack. Tekton builds push to the OpenShift internal registry (`image-registry.openshift-image-registry.svc:5000`).

[NOTE]
====
Versions reflect the CSVs installed at the time of last pattern validation. Operators on the `stable` channel auto-update within the minor version. Pin to specific versions in subscriptions if you require change control.
====
