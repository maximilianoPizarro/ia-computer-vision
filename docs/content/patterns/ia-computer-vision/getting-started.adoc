= Getting started with AI Computer Vision

This guide walks you through deploying the AI Computer Vision pattern from a clean environment. You install the hub cluster first, verify it, then deploy the east and west spoke clusters. Each section includes verification commands with expected output so you know whether each step succeeded.

[IMPORTANT]
====
Complete each cluster installation and verification before moving to the next. The hub must be healthy and RHACM must import the spokes before cross-cluster features such as Skupper, multi-cluster observability, and the RHCL inference gateway become operational.
====

== Prerequisites

=== Cluster count and sizing

* Three Red Hat OpenShift Container Platform (OCP) 4.20 or later clusters: one hub, one east spoke, one west spoke
* Recommended AWS sizing per cluster: 3 control plane nodes (`m6a.2xlarge`) + 3 worker nodes (`m6a.2xlarge`)
* See xref:cluster-sizing.adoc[Cluster sizing] for GPU options and network requirements
* CPU-only inference is the default; GPU workers are optional

=== Network requirements

* Each cluster requires outbound internet access to pull container images from `registry.redhat.io`, `quay.io`, and operator catalogs from `registry.connect.redhat.com`
* Hub and spokes must reach each other on HTTPS (443) for RHACM agent registration, Skupper links (port 8443 by default), and RHACS sensor communication (port 8443)
* A wildcard DNS record `*.apps.<cluster_domain>` is required on each cluster (default OCP ingress provides this)
* Recommended maximum inter-cluster latency: 200 ms RTT for Skupper; 100 ms RTT for responsive Grafana federation

=== Workstation requirements

* `git` 2.x and `podman` 4.x installed
* OpenShift CLI (`oc`) 4.20 or later installed and in your PATH
* Cluster administrator `kubeconfig` for each cluster
* Red Hat pull secret (from link:https://console.redhat.com/openshift/install/pull-secret[console.redhat.com]) if you deploy clusters with the OpenShift installer

=== Operator prerequisite

Install the **Validated Patterns Operator** on each cluster before you run `./pattern.sh make install`:

. Log in to the OCP web console as a cluster administrator.
. Navigate to *Operators* -> *OperatorHub*.
. Search for *Validated Patterns Operator* and install it with default settings.
. Wait for the operator to reach `Succeeded`:

[source,bash]
----
$ oc get csv -n openshift-operators | grep patterns-operator
----

.Expected output
[source,text]
----
patterns-operator.v0.0.76   Validated Patterns Operator   0.0.76   Succeeded
----

== Preparing your environment

=== Cloning the pattern repository

[source,bash]
----
$ git clone https://github.com/maximilianoPizarro/ia-computer-vision.git
$ cd ia-computer-vision
----

=== Configuring secrets

The pattern uses Vault with auto-generated secrets for most components. Copy the template and review which fields require manual input:

[source,bash]
----
$ cp values-secret.yaml.template ~/values-secret-ia-computer-vision.yaml
----

The following table explains each secret and whether you need to provide a value:

[cols="2,2,3",options="header"]
|===
| Secret name | Fields | Action required

| `gitlab-credentials`
| `root-password`, `runner-token`
| **None.** Both use `onMissingValue: generate` — Vault creates them automatically.

| `rhbk-credentials`
| `admin-password`
| **None.** Auto-generated by Vault.

| `developer-hub-secrets`
| `session-secret`, `gitlab-token`
| `session-secret` is auto-generated. `gitlab-token` requires a GitLab Personal Access Token (PAT) with `api` scope. You can set it after GitLab deploys — the scaffolder will not work until you provide it.

| `maas-credentials`
| `api-key`
| **Optional.** Required only if you enable MaaS/LiteLLM model endpoints on OpenShift AI. Leave the placeholder if you do not use external model APIs.

| `quay-credentials`
| `dockerconfigjson`
| **Optional.** Required only if you push images to the pattern Quay registry. Leave the placeholder for read-only usage.
|===

[TIP]
====
You can skip editing `~/values-secret-ia-computer-vision.yaml` entirely for a first deployment. Vault auto-generates the critical secrets (GitLab root password, Keycloak admin password, Developer Hub session secret). You can update `gitlab-token` and optional fields after the hub is running.
====

For details on how Vault and ESO manage secrets in Validated Patterns, see link:https://validatedpatterns.io/learn/secrets-management-in-the-validated-patterns-framework/[Validated Patterns secrets management].

=== Setting the active cluster context

Log in to the hub cluster:

[source,bash]
----
$ oc login --token=<hub_token> --server=https://api.<hub_domain>:6443
----

Verify you are on the correct cluster:

[source,bash]
----
$ oc whoami --show-server
----

.Expected output
[source,text]
----
https://api.<hub_domain>:6443
----

== Installing the hub cluster

.Prerequisites
* Validated Patterns Operator installed on the hub
* Hub cluster `kubeconfig` active in your shell
* (Optional) `~/values-secret-ia-computer-vision.yaml` customized

. Verify `values-global.yaml` sets `clusterGroupName: hub` (this is the default — do not change it for the hub).

. From the repository root, run:

[source,bash]
----
$ ./pattern.sh make install
----

The command creates a Pattern custom resource, which triggers the Validated Patterns Operator to install OpenShift GitOps and deploy all applications defined in `values-hub.yaml`. This process takes 30–60 minutes depending on cluster resources and image pull speed.

[NOTE]
====
Alternatively, you can create the Pattern CR directly from the OCP console: navigate to *Operators* -> *Installed Operators* -> *Validated Patterns Operator* -> *Pattern* tab -> *Create Pattern*, set `clusterGroupName: hub` and the Git repository URL.
====

== Verifying the hub installation

Allow 30–60 minutes for all operators and OpenShift GitOps applications to reach a healthy state. Run each verification in order.

=== Step 1: Verify OpenShift GitOps applications

[source,bash]
----
$ oc get applications.argoproj.io -n vp-gitops
----

.Expected output
[source,text]

[IMPORTANT]
====
If any application shows `Degraded` or `Missing`, do not proceed. Check the xref:troubleshooting.adoc[Troubleshooting] guide. Common causes include:

* **OutOfSync + Healthy**: Normal during initial install. OpenShift GitOps reconciles automatically; wait 5–10 minutes and recheck.
* **Progressing**: An operator is still installing its operand. Wait for the CSV to reach `Succeeded`.
* **Degraded**: A resource failed to apply. Run `oc describe application <app_name> -n vp-gitops` and check the `Message` field for the specific error.
* **Missing**: A CRD required by the application is not yet installed (for example, `ObjectBucketClaim` for Quay requires OpenShift Data Foundation). Check operator prerequisites.
====

== Importing spoke clusters into RHACM

This step is required before you install the spoke patterns. RHACM on the hub must know about the east and west clusters.

[IMPORTANT]
====
The Validated Patterns Operator does not auto-import spokes. You must import them into RHACM and apply the `clusterGroup` label so that RHACM policies and GitOps applications propagate correctly.
====

=== Option A: Import via RHACM console (recommended)

. On the hub, open the RHACM console: `https://multicloud-console.apps.<hub_domain>`
. Navigate to *Infrastructure* -> *Clusters* -> *Import cluster*.
. Provide the spoke cluster name (for example, `east-cluster`) and select *Import*.
. RHACM generates an import command. Run it on the spoke cluster as `kubeadmin`.
. After import succeeds, apply the cluster group label:

[source,bash]
----
$ oc label managedcluster <east_cluster_name> clusterGroup=east --overwrite
----

=== Option B: Import via CLI

[source,bash]
----
$ cat <<EOF | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1
kind: ManagedCluster
metadata:
  name: <east_cluster_name>
  labels:
    clusterGroup: east
spec:
  hubAcceptsClient: true
EOF
----

=== Verify import status

[source,bash]
----
$ oc get managedcluster
----

.Expected output
[source,text]
----
NAME                  HUB ACCEPTED   MANAGED CLUSTER URLS                                           JOINED   AVAILABLE   AGE
local-cluster         true           https://api.<hub_domain>:6443                                   True     True        60m
<east_cluster_name>   true           https://api.<east_domain>:6443                                  True     True        5m
----

Repeat for the west cluster with `clusterGroup=west`.

== Installing the east spoke

.Prerequisites
* Hub cluster verified and healthy (all applications Synced/Healthy)
* East cluster imported into RHACM with `clusterGroup=east` label
* Validated Patterns Operator installed on the east cluster
* East cluster `kubeconfig` active in your shell

. Log in to the east cluster:

[source,bash]
----
$ oc login --token=<east_token> --server=https://api.<east_domain>:6443
----

. Install the east spoke:

[source,bash]
----
$ export TARGET_CLUSTERGROUP=east
$ ./pattern.sh make install
----

Allow 15–30 minutes for spoke applications to sync.

== Installing the west spoke

Repeat the east spoke procedure on the west cluster:

. Ensure the west cluster is imported into RHACM with `clusterGroup=west`.
. Log in to the west cluster:

[source,bash]
----
$ oc login --token=<west_token> --server=https://api.<west_domain>:6443
----

. Install:

[source,bash]
----
$ export TARGET_CLUSTERGROUP=west
$ ./pattern.sh make install
----

. Verify using the same commands as the east spoke section above.

== Verifying end-to-end operation

Perform these checks from the hub cluster after both spokes are healthy and imported.

=== Verify RHCL Gateway and HTTPRoute

[source,bash]
----
$ oc get gateway -n neuroface-gateway-system
$ oc get httproute -n neuroface-gateway-system -o wide
----

.Expected: Gateway `Programmed` condition is `True`, HTTPRoute lists both east and west backend references.

=== Test inference through the hub gateway

[source,bash]
----
$ curl -sk -o /dev/null -w '%{http_code}\n' \
  https://neuroface-cv.apps.<hub_domain>/api/ppe/status
----

.Expected output
[source,text]
----
200
----

=== Validate 50/50 load balancing

Send 20 requests and observe which spoke responds:

[source,bash]
----
$ for i in $(seq 1 20); do
    curl -sk https://neuroface.apps.<hub_domain>/health 2>/dev/null | head -c 80
    echo
  done
----

Verify in Grafana or Kiali that traffic splits approximately evenly between east and west backends. Open Grafana at:

[source,text]
----
https://grafana.apps.<hub_domain>
----

=== Open the NeuroFace UI

[source,text]
----
https://neuroface.apps.<hub_domain>
----

=== Open the Showroom lab guide

[source,text]
----
https://showroom-showroom.apps.<hub_domain>
----

Log in with `user1` / `Welcome123!` to access the embedded terminal and lab exercises.
