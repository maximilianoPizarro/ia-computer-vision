= Getting started with AI Computer Vision

This guide walks you through deploying the AI Computer Vision pattern from a clean environment. Install the east and west spoke clusters first, collect their API tokens, then install the hub last so RHACM can auto-import both spokes. Each section includes verification commands with expected output so you know whether each step succeeded.

[IMPORTANT]
====
Complete each cluster installation and verification before moving to the next. The hub imports spokes automatically when you provide spoke API tokens in `~/values-secret-ia-computer-vision.yaml`. Cross-cluster features such as Skupper, multi-cluster observability, RHACS Secured Cluster registration, and the RHCL inference gateway require imported spokes.
====

== Prerequisites

=== Cluster count and sizing

* Three Red Hat OpenShift Container Platform (OCP) 4.20 or later clusters: one hub, one east spoke, one west spoke
* Recommended AWS sizing per cluster: 3 control plane nodes (`m6a.2xlarge`) + 3 worker nodes (`m6a.2xlarge`)
* See link:../cluster-sizing/[Cluster sizing] for GPU options and network requirements
* CPU-only inference is the default; GPU workers are optional

=== Network requirements

* Each cluster requires outbound internet access to pull container images from `registry.redhat.io`, `quay.io`, and operator catalogs from `registry.connect.redhat.com`
* Hub and spokes must reach each other on HTTPS (443) for RHACM agent registration, Skupper links (port 8443 by default), and RHACS sensor communication (port 8443)
* A wildcard DNS record `*.apps.<cluster_domain>` is required on each cluster (default OCP ingress provides this)
* Recommended maximum inter-cluster latency: 200 ms RTT for Skupper; 100 ms RTT for responsive Grafana federation

=== Workstation requirements

* `git` 2.x and `podman` 4.x installed
* OpenShift CLI (`oc`) 4.20 or later installed and in your PATH
* Cluster administrator `kubeconfig` for each cluster
* Red Hat pull secret (from link:https://console.redhat.com/openshift/install/pull-secret[console.redhat.com]) if you deploy clusters with the OpenShift installer

=== Operator prerequisite

Install the **Validated Patterns Operator** on each cluster before you run `./pattern.sh make install`:

. Log in to the OCP web console as a cluster administrator.
. Navigate to *Operators* -> *OperatorHub*.
. Search for *Validated Patterns Operator* and install it with default settings.
. Wait for the operator to reach `Succeeded`:

[source,bash]
----
$ oc get csv -n openshift-operators | grep patterns-operator
----

.Expected output
[source,text]
----
patterns-operator.v0.0.76   Validated Patterns Operator   0.0.76   Succeeded
----

== Preparing your environment

=== Cloning the pattern repository

[source,bash]
----
$ git clone https://github.com/maximilianoPizarro/ia-computer-vision.git
$ cd ia-computer-vision
----

=== Configuring secrets

The pattern uses Vault with auto-generated secrets for most components. Copy the template and review which fields require manual input:

[source,bash]
----
$ cp values-secret.yaml.template ~/values-secret-ia-computer-vision.yaml
----

The following table explains each secret and whether you need to provide a value:

[cols="2,2,3",options="header"]
|===
| Secret name | Fields | Action required

| `gitlab-credentials`
| `root-password`, `runner-token`
| **None.** Both use `onMissingValue: generate` — Vault creates them automatically.

| `rhbk-credentials`
| `admin-password`
| **None.** Auto-generated by Vault.

| `developer-hub-secrets`
| `session-secret`, `gitlab-token`
| `session-secret` is auto-generated. `gitlab-token` requires a GitLab Personal Access Token (PAT) with `api` scope. You can set it after GitLab deploys — the scaffolder will not work until you provide it.

| `maas-credentials`
| `api-key`
| **Optional.** Required only if you enable MaaS/LiteLLM model endpoints on OpenShift AI. Leave the placeholder if you do not use external model APIs.

| `quay-credentials`
| `dockerconfigjson`
| **Optional.** Required only if you push images to the pattern Quay registry. Leave the placeholder for read-only usage.

| `spoke-credentials`
| `east-token`, `east-api-url`, `west-token`, `west-api-url`
| **Required for hub install.** Set after east and west spokes are running. Use a cluster-admin token from each spoke and its API server URL. The `acm-hub-spoke` chart (wave 6) reads these from Vault and auto-imports both clusters into RHACM.
|===

[IMPORTANT]
====
For automated spoke import, install east and west *before* the hub and fill `spoke-credentials` in `~/values-secret-ia-computer-vision.yaml` before you run `./pattern.sh make install` on the hub.
====

[TIP]
====
You can skip editing optional secrets for a first deployment. Vault auto-generates GitLab, Keycloak, and Developer Hub secrets. You must provide `spoke-credentials` before hub install unless you import spokes manually (see xref:#importing-spoke-clusters-into-rhACM[Importing spoke clusters into RHACM]).
====

For details on how Vault and ESO manage secrets in Validated Patterns, see link:https://validatedpatterns.io/learn/secrets-management-in-the-validated-patterns-framework/[Validated Patterns secrets management].

=== Setting the active cluster context

Log in to the cluster you are installing (east, west, or hub):

[source,bash]
----

[source,bash]
----
$ oc whoami --show-server
----

.Expected output
[source,text]
----
https://api.<cluster_domain>:6443

== Installing the east spoke

.Prerequisites
* Validated Patterns Operator installed on the east cluster
* East cluster `kubeconfig` active in your shell

. Log in to the east cluster:

[source,bash]
----
$ oc login --token=<east_token> --server=https://api.<east_domain>:6443
----

. Install the east spoke:

[source,bash]
----
$ export TARGET_CLUSTERGROUP=east
$ ./pattern.sh make install
----

Allow 15–30 minutes for spoke applications to sync.

=== Collect the east spoke API token

Generate a long-lived token for RHACM auto-import on the hub. The pattern automatically creates a `acm-import` ServiceAccount with `cluster-admin` in `kube-system` (via the `platform-users` chart at wave 0), so you only need to generate the token:

[source,bash]
----
$ oc create token -n kube-system acm-import --duration=87600h
$ oc whoami --show-server
----

Record the token and API URL (`https://api.<east_domain>:6443`) in `~/values-secret-ia-computer-vision.yaml` under `spoke-credentials` as `east-token` and `east-api-url`.

$ oc get pods -n ztunnel

.Expected: `istiod` running in `istio-system`, ztunnel DaemonSet pods running on each node.

[source,bash]

$ oc get listener,connector -n service-interconnect

== Installing the west spoke

Repeat the east spoke procedure on the west cluster:

. Log in to the west cluster:

[source,bash]
----
$ oc login --token=<west_token> --server=https://api.<west_domain>:6443
----

. Install:

[source,bash]
----
$ export TARGET_CLUSTERGROUP=west
$ ./pattern.sh make install
----

. Verify using the same commands as the east spoke section above.

. Collect the west spoke API token (using the same `acm-import` SA procedure as east) and API URL. Add them to `spoke-credentials` as `west-token` and `west-api-url` in `~/values-secret-ia-computer-vision.yaml`.

== Installing the hub cluster

.Prerequisites
* Validated Patterns Operator installed on the hub
* East and west spokes installed and healthy
* `spoke-credentials` filled in `~/values-secret-ia-computer-vision.yaml`
* Hub cluster `kubeconfig` active in your shell

. Log in to the hub cluster:

[source,bash]
----
$ oc login --token=<hub_token> --server=https://api.<hub_domain>:6443
----

. Verify `values-global.yaml` sets `clusterGroupName: hub` (this is the default — do not change it for the hub).

. From the repository root, run:

[source,bash]
----
$ ./pattern.sh make install
----

The command creates a Pattern custom resource, which triggers the Validated Patterns Operator to install OpenShift GitOps and deploy all applications defined in `values-hub.yaml`. This process takes 30–60 minutes depending on cluster resources and image pull speed.

=== Alternative: Pattern CR via OCP Console

Instead of `./pattern.sh make install`, create the Pattern CR from the console: *Operators* -> *Installed Operators* -> *Validated Patterns Operator* -> *Pattern* tab -> *Create Pattern*.

For **spoke** clusters (east or west), use a minimal CR without `extraParameters`:

[source,yaml]
----
apiVersion: gitops.hybrid-cloud-patterns.io/v1alpha1
kind: Pattern
metadata:
  name: ia-computer-vision
  namespace: openshift-operators
spec:
  clusterGroupName: east    # <1>
  gitSpec:
    targetRepo: https://github.com/maximilianoPizarro/ia-computer-vision.git
    targetRevision: main
  multiSourceConfig:
    enabled: true
    clusterGroupChartVersion: "0.9.*"
    helmRepoUrl: https://charts.validatedpatterns.io
----
<1> Replace with `west` for the west spoke.

For the **hub** cluster, add `extraParameters` with the spoke tokens collected during spoke installation:

[source,yaml]
----
apiVersion: gitops.hybrid-cloud-patterns.io/v1alpha1
kind: Pattern
metadata:
  name: ia-computer-vision
  namespace: openshift-operators
spec:
  clusterGroupName: hub
  gitSpec:
    targetRepo: https://github.com/maximilianoPizarro/ia-computer-vision.git
    targetRevision: main
  multiSourceConfig:
    enabled: true
    clusterGroupChartVersion: "0.9.*"
    helmRepoUrl: https://charts.validatedpatterns.io
  extraParameters:
    - name: spokeCredentials.mode
      value: inline
    - name: spokeCredentials.clusters.east.token
      value: "<EAST_SA_TOKEN>"          # <1>
    - name: spokeCredentials.clusters.east.apiUrl
      value: "https://api.<EAST_DOMAIN>:6443"   # <2>
    - name: spokeCredentials.clusters.west.token
      value: "<WEST_SA_TOKEN>"          # <3>
    - name: spokeCredentials.clusters.west.apiUrl
      value: "https://api.<WEST_DOMAIN>:6443"   # <4>
----
<1> Long-lived cluster-admin token from east (SA created automatically by the pattern): `oc create token -n kube-system acm-import --duration=87600h`
<2> East API server URL from `oc whoami --show-server`
<3> Long-lived token from west
<4> West API server URL

[IMPORTANT]
====
Replace all `<...>` placeholders with your actual cluster values. Never commit tokens to Git.
====

[IMPORTANT]
====
If any application shows `Degraded` or `Missing`, do not proceed. Check the link:../troubleshooting/[Troubleshooting] guide. Common causes include:

* **OutOfSync + Healthy**: Normal during initial install. OpenShift GitOps reconciles automatically; wait 5–10 minutes and recheck.
* **Progressing**: An operator is still installing its operand. Wait for the CSV to reach `Succeeded`.
* **Degraded**: A resource failed to apply. Run `oc describe application <app_name> -n vp-gitops` and check the `Message` field for the specific error.
* **Missing**: A CRD required by the application is not yet installed (for example, `ObjectBucketClaim` for Quay requires OpenShift Data Foundation). Check operator prerequisites.
====

[[importing-spoke-clusters-into-rhACM]]
== Importing spoke clusters into RHACM

The `acm-hub-spoke` chart (OpenShift GitOps wave 6) auto-imports east and west when `spoke-credentials` are present in Vault. No manual RHACM console steps are required when you follow the hub-last install order above.

=== How auto-import works

. You store `east-token`, `east-api-url`, `west-token`, and `west-api-url` in `~/values-secret-ia-computer-vision.yaml`.
. The Validated Patterns Operator loads them into Vault during hub install (wave 2).
. External Secrets Operator projects them into the `spoke-credentials` Secret in `openshift-gitops`.
. The `acm-hub-spoke` CronJob creates `ManagedCluster`, `auto-import-secret`, and `KlusterletAddonConfig` resources for each spoke.
. RHACM installs the klusterlet agent on each spoke and registers the cluster with labels `clusterGroup=east|west`.
. The existing `acs-init-bundle-sync` chart (wave 7) pushes ACS init bundles to imported spokes via `ManagedClusterAction`.

=== Verify auto-import status

[source,bash]
----
$ oc get managedcluster
$ oc get cronjob acm-spoke-auto-import -n openshift-gitops
$ oc logs -n openshift-gitops -l job-name --tail=50 2>/dev/null | tail -20
----

.Expected output
[source,text]
----
NAME            HUB ACCEPTED   MANAGED CLUSTER URLS                     JOINED   AVAILABLE   AGE
local-cluster   true           https://api.<hub_domain>:6443            True     True        60m
east            true           https://api.<east_domain>:6443           True     True        10m
west            true           https://api.<west_domain>:6443           True     True        10m
----

=== Option A: Manual import via RHACM console

Use this only if you cannot provide spoke tokens in Vault (for example, a sandbox without the hub-last workflow).

. On the hub, open the RHACM console: `https://multicloud-console.apps.<hub_domain>`
. Navigate to *Infrastructure* -> *Clusters* -> *Import cluster*.
. Provide the spoke cluster name (`east` or `west`) and select *Import*.
. RHACM generates an import command. Run it on the spoke cluster as `kubeadmin`.
. After import succeeds, apply the cluster group label:

[source,bash]
----
$ oc label managedcluster east clusterGroup=east --overwrite
$ oc label managedcluster west clusterGroup=west --overwrite
----

=== Option B: Manual import via CLI

[source,bash]
----
$ cat <<EOF | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1
kind: ManagedCluster
metadata:
  name: east
  labels:
    clusterGroup: east
spec:
  hubAcceptsClient: true
EOF
----

Repeat for `west` with `clusterGroup=west`.
