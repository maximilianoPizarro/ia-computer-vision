= Getting started with AI Computer Vision

This guide walks you through deploying the AI Computer Vision Validated Pattern from a clean environment. You install the hub cluster first, verify it, then install and verify the east and west spoke clusters.

[IMPORTANT]
====
Complete each cluster installation before moving to the next. The hub must register spokes in Red Hat Advanced Cluster Management (RHACM) before cross-cluster features such as Skupper and multi-cluster observability become fully operational.
====

== Prerequisites

Before you begin, confirm that you meet these requirements.

=== Cluster count and sizing

* Three OCP 4.20 or later clusters: one hub, one east spoke, and one west spoke
* Recommended AWS sizing per cluster: 3 control plane nodes and 3 worker nodes of type `m6a.2xlarge` (see xref:cluster-sizing.adoc[Cluster sizing])
* CPU-only inference is the default; GPU workers are optional for accelerated inference on spokes

=== Network requirements

* Each cluster requires outbound internet access to pull container images and operator catalogs
* Hub and spokes must reach each other on ports required by RHACM, Skupper, and RHACS Central registration
* A wildcard DNS record or default `*.apps.<cluster_domain>` ingress is required on each cluster

=== Workstation requirements

* `git` and `podman` installed on your workstation
* Cluster administrator `kubeconfig` for each cluster (hub, east, west)
* OpenShift CLI (`oc`) installed and configured
* Red Hat pull secret available if you deploy clusters with the OpenShift installer

=== Operator prerequisite

Install the **Validated Patterns Operator** on each cluster before you run `./pattern.sh make install`:

. Log in to the OCP web console as a cluster administrator.
. Navigate to *Operators* → *OperatorHub*.
. Search for *Validated Patterns Operator* and install it into the default namespace.

== Preparing your environment

=== Cloning the pattern repository

[source,bash]
----
$ git clone https://github.com/maximilianoPizarro/ia-computer-vision.git
$ cd ia-computer-vision
----

=== Configuring secrets

Copy the secrets template and customize optional values:

[source,bash]
----
$ cp values-secret.yaml.template ~/values-secret-ia-computer-vision.yaml
----

Edit `~/values-secret-ia-computer-vision.yaml` to set optional credentials such as GitLab scaffolder tokens or Quay push credentials. Vault auto-generates most secrets when fields use `onMissingValue: generate`.

For details, see link:https://validatedpatterns.io/learn/secrets-management-in-the-validated-patterns-framework/[Validated Patterns secrets management].

=== Setting the active cluster context

Log in to the hub cluster before you install the hub pattern:

[source,bash]
----
$ oc login --token=<hub_token> --server=https://api.<hub_domain>:6443
----

== Installing the hub cluster

.Prerequisites
* Validated Patterns Operator installed on the hub
* Hub cluster administrator `kubeconfig` active in your shell

. Ensure `values-global.yaml` sets `clusterGroupName: hub` (this is the default).
. From the repository root, run:

+
[source,bash]
----
$ ./pattern.sh make install
----

Alternatively, create a Pattern custom resource with `clusterGroupName: hub` pointing to this repository.

The hub installation deploys operators and applications defined in `values-hub.yaml`, including RHACM, Vault, External Secrets Operator (ESO), RHACS Central, RHCL, GitLab, Developer Hub, OpenShift AI, Quay, Keycloak, and Showroom.

Run the following checks after the installation completes. Allow 30–60 minutes for all operators and OpenShift GitOps applications to reach a healthy state.

[source,bash]
----

----

== Installing the east spoke

.Prerequisites
* Hub cluster verified and healthy
* Validated Patterns Operator installed on the east cluster
* East cluster administrator `kubeconfig` active in your shell

. Log in to the east cluster:

+
[source,bash]
----
$ oc login --token=<east_token> --server=https://api.<east_domain>:6443
----

. Install the east spoke pattern:

+
[source,bash]
----
$ export TARGET_CLUSTERGROUP=east
$ ./pattern.sh make install
----

=== Importing the east cluster into RHACM on the hub

On the hub cluster, label the east managed cluster:

[source,bash]
----
$ oc label managedcluster <east_cluster_name> clusterGroup=east --overwrite
----

Verify import status:

[source,bash]
----
$ oc get managedcluster <east_cluster_name> -o jsonpath='{.status.conditions[?(@.type=="ManagedClusterConditionAvailable")].status}{"\n"}'
----

.Expected output
[source,text]
----
True
----

== Installing the west spoke

.Prerequisites
* East spoke verified and imported into RHACM
* Validated Patterns Operator installed on the west cluster
* West cluster administrator `kubeconfig` active in your shell

. Log in to the west cluster:

+
[source,bash]
----
$ oc login --token=<west_token> --server=https://api.<west_domain>:6443
----

. Install the west spoke pattern:

+
[source,bash]
----
$ export TARGET_CLUSTERGROUP=west
$ ./pattern.sh make install
----

. Label the west managed cluster on the hub:

+
[source,bash]
----
$ oc label managedcluster <west_cluster_name> clusterGroup=west --overwrite
----

== Verifying end-to-end operation

Perform these checks from the hub cluster or your workstation after both spokes are healthy and imported.

=== Verifying the RHCL HTTPRoute

[source,bash]
----
$ oc get httproute -n neuroface-gateway-system
$ oc get gateway -n neuroface-gateway-system
----

.Expected output (sample)
[source,text]
----
NAME                  HOSTNAMES                              AGE
neuroface-httproute   ["neuroface.apps.<hub_domain>"]        15m
----

=== Testing inference through the hub gateway

[source,bash]
----
$ curl -sk -o /dev/null -w '%{http_code}\n' \
  https://neuroface-cv.apps.<hub_domain>/api/ppe/status
----

.Expected output
[source,text]
----
200
----

=== Validating 50/50 load balancing

Send multiple requests and inspect backend distribution in RHCL/Kuadrant metrics or NeuroFace gateway logs:

[source,bash]
----
$ for i in $(seq 1 20); do
    curl -sk https://neuroface.apps.<hub_domain>/health | head -c 80
    echo
  done
----

[NOTE]
====
Exact response bodies depend on NeuroFace configuration. Confirm in Grafana or Kiali that traffic splits evenly between east and west backend references on the HTTPRoute.
====

=== Opening the NeuroFace UI

Open the following URL in a browser:

[source,text]
----
https://neuroface.apps.<hub_domain>
----

You should see the NeuroFace application interface. If authentication is enabled, use your workshop or cluster credentials.
