= Scaffolding and secrets management

This document explains how the **AI Computer Vision software template** orchestrates operators on the hub and spokes, and how **HashiCorp Vault** with the **External Secrets Operator (ESO)** delivers credentials to OpenShift without storing secret values in Git.

image::scaffolding-vault-eso-flow.png[Scaffolding and Vault/ESO flows â€” operator interaction and expected results,align="center"]

== AI Computer Vision software template

=== What it is

The software template `ai-computer-vision` in Red Hat Developer Hub (RHDH) is a Backstage *Template* resource. It gives workshop users a self-service form to provision a personal **NeuroFace** stack on a chosen spoke (`east` or `west`) without opening tickets to platform teams.

The template lives in the pattern Git repository as Helm-managed catalog data:

* Manifest: `charts/all/developer-hub/files/software-templates/ai-computer-vision.yaml`
* Skeleton: `charts/all/developer-hub/files/software-templates/ai-cv-skeleton/`

=== Operators and components involved

[cols="2,4",options="header"]
|===
| Component | Role in the scaffolding journey

| Validated Patterns Operator
| Installs hub charts (Developer Hub, GitLab, OpenShift GitOps, ACM hub-spoke) and injects cluster-specific domains into the template form.

| Red Hat Developer Hub (Scaffolder)
| Runs template steps: fetch skeleton, publish to GitLab, register catalog entity. Uses GitLab integration backed by a PAT stored in Vault.

| GitLab Operator
| Hosts GitLab on the hub. Repositories land in group `ws-workshop` with name `neuroface-<owner>`.

| OpenShift GitOps (Argo CD)
| Hosts ApplicationSet `user-neuroface-apps` in `vp-gitops`. Discovers new GitLab repos and creates Applications targeting spoke clusters.

| RHACM GitOpsCluster
| Registers managed clusters `east` and `west` with the hub Argo CD instance so ApplicationSet destinations `name: east|west` can sync.

| Spoke platform charts (`spoke-neuroface`, `spoke-neuroface-cv`, Service Mesh, Kuadrant)
| Internal spoke Gateway API (`neuroface-spoke-gateway`, no public Route) and Skupper tunnel to hub `neuroface.apps.<hub>` (50/50 east/west).
|===

=== Scaffolder steps (what runs when the user clicks Create)

[cols="1,3,4",options="header"]
|===
| Step | Action | Output

| `fetch:template`
| Copies `ai-cv-skeleton` and substitutes owner, spoke, domains, MaaS endpoint.

| `publish:github`

== Vault and External Secrets Operator

=== What it is

The pattern follows the link:https://validatedpatterns.io/learn/secrets-management-in-the-validated-patterns-framework/[Validated Patterns secrets management] model:

* **Vault** is the system of record for sensitive values on the hub.
* **ESO** reads Vault and materializes native Kubernetes `Secret` objects that pods and operators consume.
* **Git** contains only `ExternalSecret` custom resources and non-sensitive configuration â€” never plaintext passwords or tokens.

=== Operators and components involved

[cols="2,4",options="header"]
|===
| Component | Role

| Validated Patterns Operator
| Reads `values-secret.yaml` (or Pattern CR) and loads secret fields into Vault paths such as `secret/hub/developer-hub-secrets`.

| HashiCorp Vault (VP chart)
| Stores key-value pairs per logical secret (`gitlab-credentials`, `rhbk-credentials`, `developer-hub-secrets`, `spoke-credentials`, â€¦).

| External Secrets Operator
| Watches `ExternalSecret` CRs; authenticates to Vault via `ClusterSecretStore` (`vault-backend`).

| OpenShift GitOps
| Syncs `ExternalSecret` manifests from each subchart; ESO reconciles them continuously (`refreshInterval`).

| Workload operators / Deployments
| Mount or reference the generated Kubernetes `Secret` (`envFrom`, `secretRef`, `volumeMount`).
|===

=== End-to-end flow

. You define secrets in `values-secret.yaml.template` (copied to `~/values-secret-ia-computer-vision.yaml`, never committed).
. During hub install, the VP Operator writes fields into Vault (`onMissingValue: generate` creates random values automatically).
. Each chart declares an `ExternalSecret` pointing at a Vault path and property names.
. ESO creates or updates a Kubernetes `Secret` in the target namespace (for example `developer-hub-oidc-auth`, `gitlab-scm-token`, PostgreSQL credentials for RHBK).
. Pods and operators read the native OpenShift secret â€” they never talk to Vault directly.

=== Minio credentials: auto-sync from GitLab to Vault

The `ppe-model-seed` Job (in the `hub-interconnect` chart) runs as a PostSync hook after GitLab is deployed. In addition to uploading the YOLO model to the GitLab Minio bucket, it syncs the Minio `accesskey` and `secretkey` from the `gitlab-minio-secret` (in `gitlab-system`) to Vault at path `secret/data/hub/minio-credentials`.

This allows spoke clusters to read model storage credentials through ESO without manual Vault population.

Flow:

. Job reads `gitlab-minio-secret` (created by the GitLab Operator)
. Job authenticates to Vault via Kubernetes auth (`secret-manager` role)
. Job writes `accesskey` and `secretkey` to `secret/data/hub/minio-credentials`
. Spoke ESO `ExternalSecret` reads from that Vault path and creates a local secret for the YOLO PPE serving pod

=== YOLO PPE model storage (spoke-neuroface-cv)

The `spoke-neuroface-cv` chart uses `useVault: true` by default to fetch Minio credentials from Vault via ESO:

[source,yaml]
----
# spoke-neuroface-cv/values.yaml
modelStorage:
  useVault: true
  vaultKey: secret/data/hub/minio-credentials
  endpoint: http://minio-hub.service-interconnect.svc:9000
----

When `useVault` is true, the chart creates an `ExternalSecret` that projects the credentials into a Kubernetes Secret consumed by the YOLO PPE serving deployment. The Minio endpoint resolves through a Skupper Connector that tunnels to the hub GitLab Minio service.

=== Keycloak realms provisioned by rhbk-iam

The `rhbk-iam` chart provisions Keycloak realms with workshop users (`user1` through `userN`). Three realms are configured in `values-hub.yaml`:

[cols="2,3,3",options="header"]
|===
| Realm | Display name | Purpose

| `neuroface`
| NeuroFace API
| Per-user scaffolded NeuroFace OIDC (biometric 2FA)

| `maas`
| Model as a Service
| API protection for the MaaS gateway

| `cv`
| Computer Vision
| OIDC protection for `neuroface-cv.apps.<hub_domain>` route
|===

Each realm is created with a `KeycloakRealmImport` CR and receives workshop users with password `Welcome123!`. The SSO endpoint is `https://sso.apps.<hub_domain>` (configured via `keycloak.hostname: sso` in the `rhbk` chart).

=== Example: Developer Hub GitLab token

[source,yaml]
----
# values-secret.yaml (user file, not in Git)
secrets:
  - name: developer-hub-secrets
    fields:
      - name: gitlab-token
        value: "<GitLab PAT with api scope>"
----

[source,yaml]
----

    name: vault-backend
    kind: ClusterSecretStore
  data:
    - secretKey: token
      remoteRef:
        key: secret/data/hub/developer-hub-secrets
        property: gitlab-token
----

=== Example: RHBK database password

Vault path `secret/hub/rhbk-credentials` holds `admin-password` and `db-password`. ESO projects them into secrets consumed by the RHBK and PostgreSQL operands. If Vault is empty, `ExternalSecret` stays in `SecretSyncedError` and RHBK pods show `CreateContainerConfigError`.

=== AI Gateway platform key for Lightspeed and NeuroFace chat

The pattern implements a single-proxy model where all AI calls go through the central `ai-gateway` on the hub.

1. **Vault auto-generation**: A 32-character `platformApiKey` is auto-generated in Vault at `secret/hub/ai-gateway-platform-keys` during the hub install.
2. **AI Gateway**: An `ExternalSecret` syncs this key to `platform-ai-gateway-key` in `ai-gateway-system`, where Kuadrant validates it.
3. **Internal clients**: Developer Hub (Lightspeed) and NeuroFace use their own `ExternalSecret` instances to fetch the exact same key from Vault, injecting it into `VLLM_API_KEY` (for Lightspeed) or `NEUROFACE_CHAT_API_KEY` (for NeuroFace).

This ensures zero manual API key distribution for internal cluster services. Users can still request their own separate API keys for external scripts via Developer Hub using the Kuadrant DevPortal plugin.

.Chart values reference
[cols="1,2",options="header"]
|===
| Value | Description

| `vault.platformApiKeyPath`
| Path to the key in Vault (default: `secret/data/hub/ai-gateway-platform-keys`)

| `plugins.lightspeed.vaultPlatformApiKeyPath`
| Vault path Developer Hub uses to fetch the platform key for Lightspeed
|===
