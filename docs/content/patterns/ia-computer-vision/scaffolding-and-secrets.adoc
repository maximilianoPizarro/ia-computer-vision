= Scaffolding and secrets management

This document explains how the **AI Computer Vision software template** orchestrates operators on the hub and spokes, and how **HashiCorp Vault** with the **External Secrets Operator (ESO)** delivers credentials to OpenShift without storing secret values in Git.

image::scaffolding-vault-eso-flow.png[Scaffolding and Vault/ESO flows — operator interaction and expected results,align="center"]

== AI Computer Vision software template

=== What it is

The software template `ai-computer-vision` in Red Hat Developer Hub (RHDH) is a Backstage *Template* resource. It gives workshop users a self-service form to provision a personal **NeuroFace** stack on a chosen spoke (`east` or `west`) without opening tickets to platform teams.

The template lives in the pattern Git repository as Helm-managed catalog data:

* Manifest: `charts/all/developer-hub/files/software-templates/ai-computer-vision.yaml`
* Skeleton: `charts/all/developer-hub/files/software-templates/ai-cv-skeleton/`

=== Operators and components involved

[cols="2,4",options="header"]
|===
| Component | Role in the scaffolding journey

| Validated Patterns Operator
| Installs hub charts (Developer Hub, GitLab, OpenShift GitOps, ACM hub-spoke) and injects cluster-specific domains into the template form.

| Red Hat Developer Hub (Scaffolder)
| Runs template steps: fetch skeleton, publish to GitLab, register catalog entity. Uses GitLab integration backed by a PAT stored in Vault.

| GitLab Operator
| Hosts GitLab on the hub. Repositories land in group `ws-workshop` with name `neuroface-<owner>`.

| OpenShift GitOps (Argo CD)
| Hosts ApplicationSet `user-neuroface-apps` in `vp-gitops`. Discovers new GitLab repos and creates Applications targeting spoke clusters.

| RHACM GitOpsCluster
| Registers managed clusters `east` and `west` with the hub Argo CD instance so ApplicationSet destinations `name: east|west` can sync.

| Spoke platform charts (`spoke-neuroface`, `spoke-neuroface-cv`, Service Mesh, Kuadrant)
| Provide shared gateway (`neuroface-spoke-gateway`), PPE inference, and mesh policies consumed by per-user manifests.
|===

=== Scaffolder steps (what runs when the user clicks Create)

[cols="1,3,4",options="header"]
|===
| Step | Action | Output

| `fetch:template`
| Copies `ai-cv-skeleton` and substitutes owner, spoke, domains, MaaS endpoint.
| Local workspace with `catalog-info.yaml`, `k8s/`, `docs/`, `.devfile.yaml`.

| `publish:github`
| Pushes to GitLab (`gitlab.apps.<hub>/ws-workshop/neuroface-<owner>`). Uses RHDH GitLab integration (PAT from Vault via ESO).
| Remote Git repository on the hub.

| `catalog:register`
| Registers the Component/API entities in Developer Hub with plugin annotations (Kubernetes, Kiali, Tekton, GitLab, Argo CD, TechDocs).
| Catalog entity `neuroface-<owner>` visible in the portal.
|===

=== Expected result after scaffolding

Within a few minutes of a successful template run you should see:

[source,bash]
----
$ oc get application -n vp-gitops neuroface-<owner>
$ oc get pods -n neuroface-<owner> --context=<spoke>
$ oc get httproute,authpolicy -n neuroface-<owner> --context=<spoke>
----

.Expected state
[source,text]
----
Application neuroface-<owner>  → destination east or west (from k8s/target-cluster.yaml)
Pods: neuroface-backend, neuroface-frontend, rhbk-neuroface-<owner>
HTTPRoute + AuthPolicy on shared spoke gateway path /user/<owner>/
Catalog entity with working plugin tabs in Developer Hub
----

[IMPORTANT]
====
The scaffolder requires a valid GitLab PAT in Vault (`developer-hub-secrets` → `gitlab-token`) and hub Argo CD cluster registration for east/west (GitOpsCluster). See link:../getting-started/#step-5b-verify-ai-computer-vision-software-template-optional[Getting started — Step 5b] and link:../troubleshooting/#ai-cv-software-template-applicationset-does-not-deploy-to-spokes[Troubleshooting — ApplicationSet].
====

== Vault and External Secrets Operator

=== What it is

The pattern follows the link:https://validatedpatterns.io/learn/secrets-management-in-the-validated-patterns-framework/[Validated Patterns secrets management] model:

* **Vault** is the system of record for sensitive values on the hub.
* **ESO** reads Vault and materializes native Kubernetes `Secret` objects that pods and operators consume.
* **Git** contains only `ExternalSecret` custom resources and non-sensitive configuration — never plaintext passwords or tokens.

=== Operators and components involved

[cols="2,4",options="header"]
|===
| Component | Role

| Validated Patterns Operator
| Reads `values-secret.yaml` (or Pattern CR) and loads secret fields into Vault paths such as `secret/hub/developer-hub-secrets`.

| HashiCorp Vault (VP chart)
| Stores key-value pairs per logical secret (`gitlab-credentials`, `rhbk-credentials`, `developer-hub-secrets`, `spoke-credentials`, …).

| External Secrets Operator
| Watches `ExternalSecret` CRs; authenticates to Vault via `ClusterSecretStore` (`vault-backend`).

| OpenShift GitOps
| Syncs `ExternalSecret` manifests from each subchart; ESO reconciles them continuously (`refreshInterval`).

| Workload operators / Deployments
| Mount or reference the generated Kubernetes `Secret` (`envFrom`, `secretRef`, `volumeMount`).
|===

=== End-to-end flow

. You define secrets in `values-secret.yaml.template` (copied to `~/values-secret-ia-computer-vision.yaml`, never committed).
. During hub install, the VP Operator writes fields into Vault (`onMissingValue: generate` creates random values automatically).
. Each chart declares an `ExternalSecret` pointing at a Vault path and property names.
. ESO creates or updates a Kubernetes `Secret` in the target namespace (for example `developer-hub-oidc-auth`, `gitlab-scm-token`, PostgreSQL credentials for RHBK).
. Pods and operators read the native OpenShift secret — they never talk to Vault directly.

=== Example: Developer Hub GitLab token

[source,yaml]
----
# values-secret.yaml (user file, not in Git)
secrets:
  - name: developer-hub-secrets
    fields:
      - name: gitlab-token
        value: "<GitLab PAT with api scope>"
----

[source,yaml]
----
# ExternalSecret (committed in Git — charts/all/developer-hub/templates/...)
spec:
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  data:
    - secretKey: token
      remoteRef:
        key: secret/data/hub/developer-hub-secrets
        property: gitlab-token
----

.Expected result
[source,text]
----
Kubernetes Secret gitlab-scm-token in vp-gitops (synced by ESO)
ApplicationSet scmProvider can list ws-workshop GitLab projects
Scaffolder publish step succeeds for workshop users
----

=== Example: RHBK database password

Vault path `secret/hub/rhbk-credentials` holds `admin-password` and `db-password`. ESO projects them into secrets consumed by the RHBK and PostgreSQL operands. If Vault is empty, `ExternalSecret` stays in `SecretSyncedError` and RHBK pods show `CreateContainerConfigError`.

=== Modes for spoke credentials

[cols="1,3",options="header"]
|===
| Mode | When to use

| `vault`
| Default VP flow — tokens in Vault `spoke-credentials`, ESO injects into ACM import.

| `inline`
| Pattern CR `extraParameters` pass tokens directly (common in sandboxes).

| `secret`
| You pre-create the Kubernetes `Secret` manually.
|===
