= Troubleshooting AI Computer Vision

This reference lists common problems during pattern installation and operation. Each entry includes symptoms, probable cause, diagnostic commands, and resolution steps.

== OpenShift GitOps application Degraded or Unknown

=== Symptom

One or more applications in `vp-gitops` show `Degraded`, `Unknown`, or `Missing` health status. The `ia-computer-vision-hub` parent application may show `Degraded` even though individual apps appear healthy.

=== Probable cause

* Operator CSV not yet `Succeeded` — the application synced before the operator was ready
* Duplicate `OperatorGroup` in the target namespace (`TooManyOperatorGroups`)
* Stale subscription referencing a deleted or renamed package
* Sync wave ordering — downstream app synced before its prerequisite
* A custom resource requires a CRD that is not yet installed (for example, `ObjectBucketClaim` for Quay requires OpenShift Data Foundation)

=== Diagnosis

[source,bash]
----
$ oc get applications.argoproj.io -n vp-gitops
$ oc describe application <app_name> -n vp-gitops | grep -A10 "Message:"
$ oc get csv -A | grep -v Succeeded
$ oc get events -n <target_namespace> --sort-by='.lastTimestamp' | tail -20
----

=== Solution

. **Wait**: Many `OutOfSync + Healthy` states resolve within 5–10 minutes as OpenShift GitOps auto-reconciles.

. **TooManyOperatorGroups**: List OperatorGroups in the affected namespace. The pattern creates one per namespace via `clustergroup`. If a chart also creates one, delete the duplicate:

[source,bash]
----
$ oc get operatorgroups -n <namespace>
$ oc delete operatorgroup <duplicate_name> -n <namespace>
----

. **Stale subscription**: If `oc describe application` shows `ResolutionFailed` for a subscription, delete the subscription and let OpenShift GitOps recreate it:

[source,bash]
----
$ oc delete subscription.operators.coreos.com <sub_name> -n <namespace>
$ oc annotate application <app_name> -n vp-gitops argocd.argoproj.io/refresh=hard --overwrite
----

. **Force resync**:

[source,bash]
----
$ oc annotate application <app_name> -n vp-gitops argocd.argoproj.io/refresh=hard --overwrite
----

== Vault pods stuck in Init or Pending state

=== Symptom

`vault-0` pod in the `vault` namespace remains `Init:0/1`, `Pending`, or `CrashLoopBackOff` for more than 15 minutes. Applications that depend on Vault-managed secrets show `Progressing` or `Missing`.

=== Probable cause

* PVC not bound — default StorageClass missing or not provisioning
* Vault not initialized (first install)
* Vault sealed after pod restart
* `unsealvault-cronjob` in `imperative` namespace not running

=== Diagnosis

[source,bash]
----
$ oc get pods -n vault
$ oc get pvc -n vault
$ oc exec vault-0 -n vault -- vault status 2>/dev/null
$ oc get cronjob -n imperative | grep unsealvault
$ oc logs -n imperative $(oc get pods -n imperative -l job-name -o name | tail -1) 2>/dev/null
----

=== Solution

. **PVC not bound**: Verify a default StorageClass exists:

[source,bash]
----
$ oc get sc | grep default
----

If no default exists, set one:

[source,bash]
----
$ oc patch sc <storage_class_name> -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
----

. **Vault sealed**: The Validated Patterns framework uses a `unsealvault-cronjob` that runs every 5 minutes. Check its logs:

[source,bash]
----
$ oc logs -n imperative -l job-name --tail=20
----

. **Manual unseal** (if cronjob is not working):

[source,bash]
----
$ oc exec vault-0 -n vault -- vault operator unseal <unseal_key>
----

The unseal key is stored in the `vaultkeys` secret in the `imperative` namespace during first initialization.

== RHACM does not propagate to spokes

=== Symptom

East or west clusters appear as `Offline`, `Pending`, or `Unknown` in RHACM. The `clusterGroup` label is missing. Hub applications that depend on managed cluster metadata (spoke domains for RHCL, Skupper) do not receive values.

=== Probable cause

* Spoke not imported into RHACM
* Missing or incorrect `clusterGroup=east|west` label
* Klusterlet agent pods not running on the spoke
* Network connectivity issue between hub RHACM and spoke API server

=== Diagnosis

On the hub:

[source,bash]
----
$ oc get managedcluster
$ oc get managedcluster <spoke_name> -o jsonpath='{.metadata.labels}{"\n"}'
$ oc get managedcluster <spoke_name> -o jsonpath='{.status.conditions[?(@.type=="ManagedClusterConditionAvailable")].status}{"\n"}'
----

On the spoke:

[source,bash]
----
$ oc get pods -n open-cluster-management-agent
$ oc get klusterlet
----

=== Solution

. Import the spoke via RHACM console (*Infrastructure* -> *Clusters* -> *Import cluster*) or CLI.

. Apply the required label:

[source,bash]
----
$ oc label managedcluster <east_cluster_name> clusterGroup=east --overwrite
----

. If klusterlet pods are not running on the spoke, re-run the import command generated by RHACM.

. Verify connectivity: the spoke must reach the hub API server on port 6443.

== Skupper VAN does not establish cross-cluster connection

=== Symptom

`oc get site -n service-interconnect` shows `Not Connected` or no remote sites. Listeners and connectors remain `Pending`. Cross-cluster inference requests time out.

=== Probable cause

* Spoke not yet registered in RHACM (the `hub-interconnect` chart pushes `AccessToken` resources to spokes only after ACM imports `east` and `west`)
* Network firewall blocking Skupper protocol (port 8443 by default for inter-router links, port 45671 for data)
* `skupper-accesstoken-sync` PostSync job or CronJob failed before spokes were imported
* `spoke-interconnect` ArgoCD application not synced

=== Diagnosis

[source,bash]
----
$ oc get site -n service-interconnect -o yaml
$ oc get listener,connector -n service-interconnect
$ oc get pods -n service-interconnect
$ oc logs deployment/skupper-router -n service-interconnect --tail=50 2>/dev/null
$ oc get accessgrant spoke-link -n service-interconnect
$ oc get job,cronjob -n service-interconnect | grep accesstoken
$ oc logs job/skupper-accesstoken-sync-hook -n service-interconnect --tail=50 2>/dev/null
$ oc get managedclusteraction -A | grep skupper
----

Check on both hub and spoke clusters.

=== Solution

. Verify `hub-interconnect` and `spoke-interconnect` apps are `Synced`:

[source,bash]
----
$ oc get application -n vp-gitops | grep interconnect
----

. On the hub, confirm the `AccessGrant` is ready and the sync job ran after RHACM imported spokes (`acm-hub-spoke` sync wave 6 runs after `hub-interconnect` wave 5; the CronJob retries every 30 minutes until links are `Ready`):

[source,bash]
----
$ oc get accessgrant spoke-link -n service-interconnect -o jsonpath='{.status.code}{"\n"}'
$ oc get cronjob skupper-accesstoken-sync -n service-interconnect
----

. If sites show `Not Connected`, verify the spoke can reach the hub on port 8443:

[source,bash]
----
$ curl -sk https://skupper-inter-router-service-interconnect.apps.<hub_domain>:443
----

. To re-run link registration without storing tokens in Git, delete the PostSync job so Argo CD recreates it on the next sync, or wait for the next CronJob run:

[source,bash]
----
$ oc delete job skupper-accesstoken-sync-hook -n service-interconnect --ignore-not-found
----

. Verify RHACM has provided the spoke domain metadata to the hub. The RHCL and Skupper charts use `global.localClusterDomain` injected by the Validated Patterns Operator.

== Service Mesh ambient mode not applying mTLS

=== Symptom

Traffic between pods in mesh-enrolled namespaces is not encrypted. Kiali shows connections without the lock icon. `oc get ztunnel` shows no DaemonSet pods.

=== Probable cause

* `servicemeshoperator3` CSV not `Succeeded`
* `servicemesh-config` ArgoCD application not synced
* Namespace not labeled for mesh enrollment
* `istio-cni` or `ztunnel` namespace missing

=== Diagnosis

[source,bash]
----
$ oc get csv -A | grep servicemeshoperator3
$ oc get pods -n istio-system
$ oc get pods -n ztunnel
$ oc get namespace <namespace> -o jsonpath='{.metadata.labels.istio\.io/dataplane-mode}{"\n"}'
----

=== Solution

. Verify the Service Mesh operator is `Succeeded`:

[source,bash]
----
$ oc get csv -A | grep servicemeshoperator3
----

. Verify the `servicemesh-config` application deployed the Istio CR:

[source,bash]
----
$ oc get istio -n istio-system
----

. Verify the ztunnel DaemonSet is running on each node:

[source,bash]
----
$ oc get pods -n ztunnel -o wide
----

. If a namespace is not enrolled in the mesh, the `servicemesh-config` chart labels namespaces automatically. Check:

[source,bash]
----
$ oc get namespace neuroface -o jsonpath='{.metadata.labels}{"\n"}'
----

== OVMS pod in CrashLoopBackOff (model not loaded)

=== Symptom

Pods in the `neuroface-cv` namespace restart repeatedly. Inference endpoints return `503` or connection refused.

=== Probable cause

* Model artifact missing or wrong mount path
* Insufficient memory for model load (OOMKilled)
* Init container failed to download YOLO weights
* GPU requested but not available on worker nodes

=== Diagnosis

[source,bash]
----
$ oc get pods -n neuroface-cv
$ oc logs deploy/ovms -n neuroface-cv --previous 2>/dev/null
$ oc describe pod -n neuroface-cv -l app=ovms | grep -E "State:|Reason:|Exit Code:"
$ oc get events -n neuroface-cv --sort-by='.lastTimestamp' | tail -10
----

=== Solution

. **OOMKilled**: Increase memory limits in `charts/all/spoke-neuroface-cv/values.yaml`.

. **Model path error**: Check the model mount configuration. The OVMS pod expects models at the path defined in the chart values.

. **GPU not available**: If the chart requests `nvidia.com/gpu` but no GPU nodes exist, remove the GPU resource request.

. After fixing, resync the application:

[source,bash]
----
$ oc annotate application east-spoke-neuroface-cv -n vp-gitops argocd.argoproj.io/refresh=hard --overwrite
----

== RHCL HTTPRoute does not balance 50/50 between spokes

=== Symptom

All inference traffic routes to one spoke only. Grafana or access logs show no hits on the other spoke.

=== Probable cause

* Missing `clusters.east.domain` or `clusters.west.domain` in `neuroface-gateway` values
* One spoke is not registered in RHACM, so its domain metadata is not injected
* One spoke NeuroFace deployment is unhealthy — the gateway removes it from rotation
* HTTPRoute weights are misconfigured
* RHCL Gateway CR shows `Waiting for controller` — operator not yet reconciled

=== Diagnosis

[source,bash]
----
$ oc get gateway -n neuroface-gateway-system -o yaml | grep -A5 "conditions:"
$ oc get httproute -n neuroface-gateway-system -o yaml | grep -A20 "backendRefs:"
$ oc get svc -n neuroface-gateway-system
----

On each spoke, verify NeuroFace is reachable:

[source,bash]
----
$ curl -sk -o /dev/null -w '%{http_code}\n' https://neuroface.apps.<east_domain>/health
----

=== Solution

. Confirm both spokes are imported in RHACM and their domains are populated in the Helm values.

. Verify the RHCL operator CSV is `Succeeded`:

[source,bash]
----
$ oc get csv -n redhat-connectivity-link-operator | grep rhcl
----

. If the Gateway shows `Waiting for controller`, the RHCL operator may have been recently reinstalled. Wait 2–3 minutes for the controller to reconcile.

. Inspect HTTPRoute weights in `charts/all/neuroface-gateway/templates/httproute.yaml` — default is 50/50 when both spoke domains are set.

== GitLab operator fails with TooManyOperatorGroups

=== Symptom

GitLab CSV shows `Failed` with reason `TooManyOperatorGroups`. The `gitlab-operator` ArgoCD application is `Healthy` but `OutOfSync`.

=== Probable cause

The `gitlab-system` namespace has two OperatorGroups: one created by the `clustergroup` chart (via `operatorGroup: true` in `values-hub.yaml`) and one created by a previous chart revision.

=== Diagnosis

[source,bash]
----
$ oc get operatorgroups -n gitlab-system
$ oc get csv -n gitlab-system | grep gitlab
----

=== Solution

. Delete the duplicate OperatorGroup (keep only the one created by `clustergroup`):

[source,bash]
----
$ oc delete operatorgroup <duplicate_name> -n gitlab-system
----

. Delete the failed CSV and subscription to force a clean reinstall:

[source,bash]
----
$ oc delete csv <gitlab_csv_name> -n gitlab-system
$ oc delete subscription.operators.coreos.com gitlab-operator-kubernetes -n gitlab-system
----

. Force ArgoCD to resync:

[source,bash]
----
$ oc annotate application ia-computer-vision-hub -n vp-gitops argocd.argoproj.io/refresh=hard --overwrite
----

== ExternalSecret shows RemoteRef key empty (RHBK)

=== Symptom

`ExternalSecret` resources in `keycloak-system` show `spec.data[0].remoteRef.key: Required value`. The RHBK application shows `Progressing` indefinitely.

=== Probable cause

Vault has not been initialized or the unseal cronjob has not populated the secrets yet. The ExternalSecret references a Vault path that does not exist.

=== Diagnosis

[source,bash]
----
$ oc get externalsecret -n keycloak-system
$ oc exec vault-0 -n vault -- vault kv list secret/ 2>/dev/null
----

=== Solution

. Verify Vault is initialized and unsealed (see the Vault troubleshooting section above).

. Check if the `unsealvault-cronjob` has run and populated secrets:

[source,bash]
----
$ oc get cronjob -n imperative
$ oc logs -n imperative $(oc get pods -n imperative -o name | tail -1) --tail=20
----

. If secrets are not populated after Vault is healthy, trigger the imperative job manually:

[source,bash]
----
$ oc create job manual-unseal --from=cronjob/unsealvault-cronjob -n imperative
----

== Hub CPU exhaustion (pods Pending with Insufficient cpu)

=== Symptom

Multiple pods remain in `Pending` state. `oc describe pod` shows:

[source,text]
----
0/6 nodes are available: 3 Insufficient cpu, 3 node(s) had untolerated taint {node-role.kubernetes.io/master: }
----

=== Cause

The hub runs 15+ operators producing ~450 pods. With `m6a.2xlarge` workers (8 vCPU allocatable), the 3 schedulable worker nodes reach 98-99% CPU request allocation. Components with high CPU requests include:

* `rhods-dashboard` — 3.1 CPU per pod (9 containers), 2 replicas = 6.2 CPU
* ACS scanner-v4 — indexer (1.5 CPU x2), matcher (500m x2), scanner-v4-db (1 CPU)
* `istiod-openshift-gateway` — 500m
* `kube-auth-proxy` — 500m per replica

=== Solution

**Option A (recommended)**: Use `m6a.4xlarge` workers on the hub (16 vCPU, 64 GiB).

**Option B (sandbox)**: Allow scheduling on master nodes:

[source,bash]
----
$ oc adm taint nodes --all node-role.kubernetes.io/master:NoSchedule-
----

[NOTE]
====
This taint reappears when the machine config operator reconciles. For a persistent solution, set `mastersSchedulable: true` in the `Scheduler` CR.
====

**Option C**: Reduce resource requests on heavy operators:

[source,bash]
----
$ oc scale deployment rhods-dashboard -n redhat-ods-applications --replicas=1
$ oc patch central stackrox-central-services -n stackrox --type merge \
  -p '{"spec":{"scannerV4":{"indexer":{"scaling":{"replicas":1},"resources":{"requests":{"cpu":"250m"}}},"matcher":{"scaling":{"replicas":1},"resources":{"requests":{"cpu":"250m"}}}}}}'
----

== Developer Hub CrashLoopBackOff (dynamic plugin ENOENT)

=== Symptom

The `backstage-developer-hub` pod crashes in `Init:CrashLoopBackOff`. The `install-dynamic-plugins` init container logs show:

[source,text]
----
InstallException: Error while installing plugin .../backstage-community-plugin-ocm with 'npm pack':
ENOENT: no such file or directory, open '.../package.json'
----

=== Cause

The RHDH 1.10 image does not ship certain community plugins that were available in earlier versions: `ocm`, `tekton`, `topology`, `kafka`, `quay`, `acr`. The dynamic plugins ConfigMap references them with `disabled: false`.

=== Solution

Set `disabled: true` on all missing plugins in the `dynamic-plugins-rhdh` ConfigMap:

[source,bash]
----
$ oc get configmap dynamic-plugins-rhdh -n developer-hub -o yaml | \
  sed 's/backstage-community-plugin-ocm.*/&\n  disabled: true/' | \
  oc apply -f -
----

Or patch the ConfigMap using Python/YAML to set `disabled: true` on all `backstage-community-plugin-*` entries from `./dynamic-plugins/dist/` that do not exist in the image. The following plugins must be disabled in RHDH 1.10: `ocm`, `ocm-backend-dynamic`, `tekton`, `topology`, `kafka`, `kafka-backend-dynamic`, `acr`, `quay`, `quay-backend-dynamic`, `rbac`.

After patching, delete the pod to restart with the corrected config:

[source,bash]
----
$ oc delete pod -n developer-hub -l app.kubernetes.io/name=backstage
----

== OCM / Kiali plugins (ghcr.io OCI)

=== Symptom

`install-dynamic-plugins` fails pulling `backstage-community-plugin-ocm` or `backstage-community-plugin-kiali`, or sidebar routes `/ocm` / `/kiali` return 404.

=== Cause

RHDH 1.9+ loads community plugins from `oci://ghcr.io/redhat-developer/rhdh-plugin-export-overlays/...` (not from `./dynamic-plugins/dist/`). Tags must match the Backstage version bundled with RHDH (`bs_1.45.3__*` for RHDH 1.9). Network policies or registry blocks can cause 403 on ghcr.io.

=== Verify ghcr pull from the cluster

[source,bash]
----
$ oc run skopeo-test --rm -i --restart=Never \
  --image=registry.redhat.io/ubi9/skopeo:latest \
  --command -- skopeo inspect \
  docker://ghcr.io/redhat-developer/rhdh-plugin-export-overlays/backstage-community-plugin-kiali:bs_1.45.3__1.49.0
----

=== Enable / configure

In `charts/all/developer-hub/values.yaml`:

* `plugins.ocm.enabled: true` — tags `bs_1.45.3__5.11.1` (frontend), `bs_1.45.3__5.12.2` (backend package `backstage-community-plugin-ocm-backend`, not `-backend-dynamic`)
* `plugins.kiali.enabled: true` — tags `bs_1.45.3__1.49.0` (frontend), `bs_1.45.3__1.28.0` (backend)

Override registry for a Quay mirror: `plugins.registry: quay.io/maximilianopizarro/rhdh-plugins`.

=== Kiali entity tab

Components need `backstage.io/kubernetes-namespace` (and optionally `backstage.io/kubernetes-id`). Example: `neuroface-backend`, `workshop-apis-gateway`.

=== OCM prerequisites

ACM `MultiClusterHub` healthy; `ClusterRoleBinding` `backstage-ocm-plugin` bound to SA `developer-hub` (created automatically when `plugins.ocm.enabled` is true); ManagedClusters visible in ACM.

=== Automated install checks

On each `developer-hub` Argo CD sync:

* **PreSync** job `developer-hub-plugin-ghcr-readiness` — verifies ghcr.io OCI images are pullable before the Backstage pod restarts.
* **PostSync** job `developer-hub-plugin-readiness` — verifies `/healthcheck`, `/ocm`, and `/kiali` when those plugins are enabled.

Disable ghcr pre-check for air-gapped mirrors: `pluginReadiness.ghcrCheck.enabled: false` and set `plugins.registry` to your Quay mirror.

== Developer Hub CrashLoopBackOff (Lightspeed sidecars)

=== Symptom

Pod `backstage-developer-hub-*` shows `1/3` or `3/3` with `CrashLoopBackOff`. Containers `llama-stack` and/or `lightspeed-core` fail. Route returns HTTP 503.

=== Cause

Hub enables Lightspeed via `plugins.lightspeed.enabled: true` in `values-hub.yaml`. Sidecar images (`llama-stack:0.1.4`, `lightspeed-stack:0.4.0`) require a compatible `llama-stack-run.yaml`. Mismatched safety providers or missing grant fields cause startup failure. The main `backstage-backend` container is often healthy; sidecar crashes mark the whole pod unhealthy.

=== Solution

This pattern ships with Lightspeed **disabled** on the hub (`plugins.lightspeed.enabled: false` in `values-hub.yaml`). Re-enable only after validating llama-stack config and image pins on a test cluster.

If already deployed with sidecars:

[source,bash]
----
# Confirm failing containers
oc logs -n developer-hub -l rhdh.redhat.com/app=backstage-developer-hub -c llama-stack --tail=20
oc logs -n developer-hub -l rhdh.redhat.com/app=backstage-developer-hub -c lightspeed-core --tail=20

# Sync developer-hub after setting plugins.lightspeed.enabled=false in values-hub.yaml
oc patch application developer-hub -n vp-gitops --type merge \
  -p '{"operation":{"initiatedBy":{"username":"admin"},"sync":{"revision":"main"}}}'
----

== NeuroFace hub route: no healthy upstream

=== Symptom

`https://neuroface.apps.<hub_domain>/dashboard` returns HTTP 503 with body `no healthy upstream` (istio-envoy).

=== Cause

The hub gateway load-balances to `neuroface-app-east` and `neuroface-app-west` via Skupper (`service-interconnect`). Listeners on the hub show `Pending` / `No matching connectors` when spoke **links** are not `Ready`. Common triggers:

* AccessGrant `spoke-link` exhausted all redemptions (sync job retried token apply in a loop)
* Stale AccessToken on a spoke (grant URL rotated but token still references old URL)
* `hub-link` secret left on spoke after failed token apply

=== Solution

[source,bash]
----
# Hub: listener/connector pairing
oc get listener,connector -n service-interconnect | grep neuroface

# Spokes: link must be Ready
oc get link hub-link -n service-interconnect

# Check grant redemptions (hub)
oc get accessgrant spoke-link -n service-interconnect
----

If redemptions are exhausted, delete and recreate the grant (hub), then re-apply tokens on each spoke (delete `accesstoken`, `link`, and secret `hub-link` first). The hardened `skupper-accesstoken-sync` job skips re-redemption when the token is already `Ready` and increases `redemptionsAllowed` to 50.

After links are `Ready`, hub listeners should show `Ready` and the dashboard should return HTTP 200.

== Vault secrets empty after install

=== Symptom

ExternalSecrets show `SecretSyncedError` with `Secret does not exist` from Vault. RHBK PostgreSQL shows `CreateContainerConfigError: secret "postgresql-db" not found`.

=== Cause

Vault is initialized and unsealed, but `./pattern.sh make load-secrets` was never run. This happens when the pattern is installed via the OCP console Pattern CR instead of the CLI.

=== Solution

Run `./pattern.sh make load-secrets` from the repository root, or manually populate Vault:

[source,bash]
----
$ oc exec vault-0 -n vault -- sh -c '
  export VAULT_ADDR=https://127.0.0.1:8200 VAULT_SKIP_VERIFY=true
  vault kv put secret/hub/rhbk-credentials \
    admin-password="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)" \
    db-password="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)"
  vault kv put secret/hub/gitlab-credentials \
    root-password="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)" \
    runner-token="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 32)"
  vault kv put secret/hub/developer-hub-secrets \
    oidc-client-secret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 32)" \
    session-secret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 32)" \
    gitlab-token="placeholder-until-gitlab-token-setup"
  vault kv put secret/hub/ai-gateway-platform-keys \
    platformApiKey="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 32)"
  vault kv put secret/hub/workshop-registration \
    adminToken="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)"
  vault kv put secret/hub/minio-credentials \
    accesskey="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 16)" \
    secretkey="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)"
  vault kv put secret/hub/keycloak-realm-clients \
    neuroface.user1.clientSecret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)" \
    neuroface.user2.clientSecret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)" \
    maas.user1.clientSecret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)" \
    cv.user1.clientSecret="$(cat /dev/urandom | tr -dc a-zA-Z0-9 | head -c 24)"
'
----

[NOTE]
====
The `rhbk-credentials` secret must include both `admin-password` and `db-password`. If `db-password` is missing, the PostgreSQL ExternalSecret fails with `cannot find secret data for key: "db-password"`.
====

== AI CV software template: ApplicationSet does not deploy to spokes

After scaffolding `neuroface-<user>`, the GitLab repo exists but no Argo CD Application appears, or the Application shows `Unknown` cluster / fails to sync.

=== Symptom

[source,bash]
----
$ oc get gitopscluster hub-spoke-gitops -n openshift-gitops -o jsonpath='{.status.conditions[?(@.type=="PlacementResolved")].message}{"\n"}'
$ oc get secrets -n vp-gitops -l argocd.argoproj.io/secret-type=cluster -o name
----

.Problem output
[source,text]
----
Successfully resolved 0 managed clusters from placement
(no east/west cluster secrets)
----

=== Cause

The hub `GitOpsCluster` must live in the same namespace as the ACM `Placement` (`openshift-gitops`) and set `spec.argoServer.argoNamespace` to the Argo CD instance namespace (`vp-gitops` for this pattern). Without that pairing, Argo CD never registers east/west, and ApplicationSet destinations `name: east|west` cannot sync.
