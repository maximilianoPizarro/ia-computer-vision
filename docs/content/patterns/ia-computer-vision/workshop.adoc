= Workshop mode

The AI Computer Vision pattern includes workshop mode for hands-on events. Workshop mode is **enabled by default** with **30 users** (`user1` through `user30`).

== What gets provisioned

[cols="2,3,2",options="header"]
|===
| Platform | Access method | Default password

| OCP Console
| HTPasswd identity provider `workshop-users`
| `Welcome123!`

| Developer Hub
| Keycloak realm `backstage` (OIDC)
| `Welcome123!`

| GitLab
| User account + `ws-userN` group (owner)
| `Welcome123!`

| DevSpaces
| Git credentials in `userN-devspaces` namespace
| `Welcome123!`
|===

Special accounts: `platformadmin` (platform engineering group) and `admin` (workshop administrator).

=== Self-service registration

Attendees self-assign a workshop user at the registration portal (chart `workshop-registration`, sync wave 5):

[source,text]
----
https://workshop-registration.apps.<hub_domain>/
----

`POST /api/register` with `{email}` assigns the first free `userN` and redirects to Showroom with `?USER_NAME=userN`. Admin token and Keycloak credentials come from Vault via External Secrets Operator (`secret/data/hub/workshop-registration`).

Embedded terminal helpers (defined in `charts/all/showroom/templates/showroom.yaml`):

[source,bash]
----
hub-login userN          # oc login to hub API (uses HUB_API_URL)
east-login userN         # oc login to east spoke (uses spokeCredentials.clusters.east.apiUrl)
west-login userN         # oc login to west spoke
sa-login                 # read-only Showroom service account
neuroface-cv-status      # curl PPE status via neuroface-cv gateway
neuroface-cv-traffic     # inspect HTTPRoute + Skupper connectors
neuroface-spoke-status userN   # pods in neuroface-userN on spoke
maas-native-status             # MaaS tenant, CRs, and gateway hostname
argocd-mcp-status              # Argo CD MCP deployment and endpoint
----

=== Troubleshooting showroom pod

If the showroom pod is not running, check the deployment status:

[source,bash]
----
$ oc get deploy,pods -n showroom
$ oc describe pod -n showroom -l app.kubernetes.io/name=showroom
$ oc logs -n showroom deploy/showroom -c git-cloner
$ oc logs -n showroom deploy/showroom -c antora-builder
----

Common failure causes:

* **git-cloner init container fails** — the content Git repository is unreachable or the branch ref does not exist. Verify `showroom.content.repoUrl` and `showroom.content.repoRef` in `charts/all/showroom/values.yaml`.
* **antora-builder fails** — `site.yml` in the content repo is misconfigured. Check the antora-builder logs.
* **PVC not bound** — the `showroom-terminal-lab-user-home` PVC requires a default StorageClass. Verify with `oc get pvc -n showroom`.
* **Terminal image pull fails** — the `quay.io/rhpds/openshift-showroom-terminal-ocp:latest` image may not be accessible from the cluster.

== Demo routes and OIDC integration

The pattern exposes two main routes on the hub gateway:

[cols="3,2,3",options="header"]
|===
| Route | OIDC protection | Purpose

| `https://neuroface.apps.<hub_domain>`
| None (public)
| Main demo frontend and backend — no login required

| `https://neuroface-cv.apps.<hub_domain>`
| Keycloak realm `cv`
| PPE detection API — requires SSO authentication
|===

The SSO login page is served at:

[source,text]
----
https://sso.apps.<hub_domain>
----

Users authenticate with `user1` / `Welcome123!` (or `userN` for workshop participants). The SSO realm is `cv`, provisioned by the `rhbk-iam` chart.

Workshop users can create their own OIDC clients in Keycloak (roles `view-clients`, `manage-clients` on `realm-management`). The Showroom module *OIDC & Keycloak Lab* (`04b-oidc-keycloak.adoc`) walks through client_credentials curl and Swagger UI testing.

Developer Hub catalog system *IAM / OIDC Realms* exposes OpenAPI specs for realms `neuroface`, `cv`, and `maas`.

=== Cluster badge (top-right corner)

When accessing `neuroface.apps.<hub_domain>`, the frontend displays a cluster badge in the top-right corner of the page:

* **EAST** (green badge) — the request was served by the east spoke
* **WEST** (yellow badge) — the request was served by the west spoke

The badge indicates which spoke cluster is handling the current request through the 50/50 HTTPRoute weight distribution. Refresh the page to observe traffic switching between spokes.

== Module 02 — Scaffold NeuroFace with RHBK biometric OIDC

Showroom module 02 walks through the AI Computer Vision software template journey:

. Log in to Developer Hub as `userN` / `Welcome123!`
. Open **Create** → **AI Computer Vision at the Edge**
. Select target spoke (east or west) and confirm hub/spoke apps domains
. Submit — the scaffolder creates a GitLab repo in `ws-workshop/neuroface-userN`
. Argo CD ApplicationSet on the **hub** (`vp-gitops`) discovers the repo and deploys `k8s/` manifests to the selected spoke
. Verify ApplicationSet: `oc get applicationset -n vp-gitops user-neuroface-apps`

For operator interaction diagrams and expected results, see link:../scaffolding-and-secrets/[Scaffolding and secrets].

. Open your catalog entity → tab **API** → NeuroFace OpenAPI spec with **Try It Out**
. Click **Open in DevSpaces** to edit NeuroFace source
. Tab **CI** shows Tekton pipeline run status (when Tekton plugin is enabled)
. Access the main demo at `https://neuroface.apps.<hub_domain>` (no login required; 50/50 traffic split to spokes)
. Access the protected CV API at `https://neuroface-cv.apps.<hub_domain>` — SSO redirects to `https://sso.apps.<hub_domain>`, login with `userN` / `Welcome123!`
. First login: complete biometric enrollment (webcam, 3–5 images)
. Subsequent logins: password + facial 2FA verification
. Test PPE detection and MaaS chat in the NeuroFace UI
. Trigger a PPE violation (remove hardhat in webcam) — NeuroFace publishes to Kafka topic `cv.ppe.detections`
. Open **Mailpit** from the OCP console menu → verify alert email arrived
. Verify Kafka consumer: `oc logs -n mailpit deploy/ppe-kafka-mailpit -f`

== Module 09 — Native Models as a Service in OpenShift AI

Showroom module 09 walks through RHOAI 3.4 native MaaS (not the Kuadrant AI Gateway path from module 05):

. Log in to the OpenShift AI dashboard as `userN`
. Confirm **Gen AI Studio** and **Models as a Service** navigation (patched by chart `models-as-a-service`)
. Run `maas-native-status` in the Showroom terminal — Tenant Ready, gateway hostname `maas.apps.<hub_domain>`
. Create an API key in Gen AI Studio and test `curl` against `https://maas.apps.<hub_domain>/v1/chat/completions`

Charts involved: `openshift-ai-hub` (`maas.native.enabled=true`), `models-as-a-service`, `workshop-kuadrant-apis` (`maas.authorinoTls=true`).

== Module 10 — Argo CD MCP and OpenShift Lightspeed

Showroom module 10 documents the GitOps MCP stack for multi-cluster queries:

. Browse Argo CD at `https://vp-gitops-server-vp-gitops.apps.<hub_domain>/`
. Run `argocd-mcp-status` — Deployment health and in-cluster endpoint `http://argocd-mcp.argocd-mcp.svc.cluster.local:3000/mcp`
. Trace tokens: `argocd-local-users` → Vault `secret/hub/argocd-mcp-tokens` → ESO → MCP Deployment

[NOTE]
====
OpenShift Lightspeed in Developer Hub may be disabled in sandbox (`plugins.lightspeed.enabled: false`). The MCP server still runs for operator wiring. Do not reuse `spokeCredentials` ACM import tokens for Argo CD MCP — they serve different purposes.
====

Charts involved: `argocd-local-users` (hub + spokes), `argocd-mcp-spoke-export` (spokes), `argocd-mcp` (hub).

* `platform-users`
* `developer-hub`
* `gitlab-operator`
* `openshift-ai-hub`
* `models-as-a-service`
* `devspaces`
* `showroom` (`showroom.terminal.userCount`)
