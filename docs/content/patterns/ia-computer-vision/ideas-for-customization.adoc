= Ideas for customization

This document describes actionable ways to extend the AI Computer Vision pattern for production or proof-of-concept environments. Each scenario includes context, files to change, and a validation command.

== Adding a third spoke (south) for a new region

=== Context

You operate inference in three geographic regions and want a third edge cluster (south) registered in RHACM alongside east and west.

=== What to change

. Create `values-south.yaml` by copying `values-east.yaml` and replacing cluster group references:

* `clusterGroupName: south`
* `clusterGroup.name: south`
* All `clusterName` overrides: `south`
* `clusterSuffix: "-south"` in observability overrides

. Add a `managedClusterGroups.south` entry in `values-hub.yaml`:

[source,yaml]
----
managedClusterGroups:
  south:
    name: south
    acmlabels:
      - name: clusterGroup
        value: south
----

. Update `charts/all/neuroface-gateway/values.yaml` or hub overrides to include `clusters.south.domain`.

. Copy spoke charts application entries from `values-east.yaml` into the new values file.

=== Files affected

* `values-south.yaml` (new)
* `values-hub.yaml`
* `charts/all/neuroface-gateway/values.yaml`
* `charts/all/observability/values.yaml` (multi-cluster tokens)

=== Validation

[source,bash]
----
$ oc label managedcluster <south_cluster_name> clusterGroup=south --overwrite
$ oc get managedcluster <south_cluster_name> -o jsonpath='{.metadata.labels.clusterGroup}{"\n"}'
$ oc get applications.argoproj.io -n vp-gitops | grep south
----

.Expected output
[source,text]
----
south
south-spoke-neuroface          Synced        Healthy
----

== Replacing YOLO with a custom model served by OVMS

=== Context

Your organization trained a proprietary object-detection model and wants to serve it through OVMS instead of the default YOLO weights.

=== What to change

. Store your model artifacts in a PVC or S3-compatible bucket accessible from the spoke.

. Edit `charts/all/spoke-neuroface-cv/values.yaml`:

* Update `ovms.modelPath` or model server configuration to reference your model directory
* Change container image or init container logic if your model requires a custom conversion step

. Update ConfigMap or InferenceService templates if you use KServe instead of standalone OVMS.

=== Files affected

* `charts/all/spoke-neuroface-cv/values.yaml`
* `charts/all/spoke-neuroface-cv/templates/` (model mount and environment variables)
* Optional: `values-east.yaml` and `values-west.yaml` overrides for model URL

=== Validation

[source,bash]
----
$ oc get pods -n neuroface-cv
$ oc logs deploy/ovms -n neuroface-cv --tail=20
$ curl -sk -X POST https://neuroface-cv.apps.<hub_domain>/v2/models/<model_name>/infer \
  -H 'Content-Type: application/json' -d @sample-request.json
----

.Expected output
[source,text]
----
NAME                     READY   STATUS    RESTARTS   AGE
ovms-xxxxxxxxxx-xxxxx    2/2     Running   0          5m
{"detections":[...]}
----

== Enabling GPU workers on spokes

=== Context

CPU inference meets functional requirements but latency exceeds your SLA. You add GPU worker nodes on east and west spokes.

=== What to change

. Provision GPU worker nodes (for example, AWS `g4dn.xlarge` with NVIDIA T4 or `g5.2xlarge` with A10G).

. Install the NVIDIA GPU Operator on each spoke (not included by default in this pattern).

. Edit `charts/all/spoke-neuroface-cv/values.yaml`:

* Set resource limits requesting `nvidia.com/gpu: 1`
* Select a GPU-enabled OVMS or ModelMesh runtime

. Document sizing changes in `docs/content/patterns/ia-computer-vision/cluster-sizing.adoc`.

=== Files affected

* Spoke cluster machine sets or worker node pools (platform-specific)
* `charts/all/spoke-neuroface-cv/values.yaml`
* `pattern-metadata.yaml` (`external_requirements.cluster_sizing_note`)
* `overrides/values-AWS.yaml` (optional GPU node selectors)

=== Validation

[source,bash]
----
$ oc get nodes -l nvidia.com/gpu.present=true
$ oc describe pod -n neuroface-cv -l app=ovms | grep -A2 Limits
$ curl -sk https://neuroface-cv.apps.<hub_domain>/api/ppe/status
----

.Expected output
[source,text]
----
gpu-worker-east-xxx   Ready   30m
  Limits:
    nvidia.com/gpu:  1
200
----

== Integrating Red Hat Trusted Artifact Signer (RHTAS) for image signing

=== Context

Your supply chain policy requires signed container images before deployment to edge clusters.

=== What to change

. Uncomment the RHTAS, RHTPA, and cert-manager subscription entries in `values-hub.yaml`:

[source,yaml]
----
# rhtas:
#   name: rhtas-operator
#   channel: alpha
# rhtpa:
#   name: rhtpa-operator
#   channel: alpha
# cert-manager:
#   name: openshift-cert-manager-operator
#   channel: stable-v1
----

. Add Argo CD applications for TSSC charts under `charts/all/` or VP published charts.

. Configure GitLab CI/CD pipelines in `charts/all/gitlab-operator/` to sign images with RHTAS before push to Quay.

=== Files affected

* `values-hub.yaml` (subscriptions and applications)
* `charts/all/gitlab-operator/templates/` (CI pipeline ConfigMaps)
* `values-secret.yaml.template` (signing credentials if not Vault-generated)

=== Validation

[source,bash]
----
$ oc get csv -n openshift-operators | grep rhtas
$ cosign verify --certificate-identity-regexp=.* \
  --certificate-oidc-issuer-regexp=.* \
  quay.apps.<hub_domain>/<org>/<image>:<tag>
----

.Expected output
[source,text]
----
rhtas-operator.vx.x.x   Red Hat Trusted Artifact Signer   Succeeded
Verification succeeded
----

== Configuring Grafana alerts for inference latency SLA

=== Context

Your SLA requires inference p95 latency below 500 ms. You create Grafana alerts when latency exceeds the threshold.

=== What to change

. Edit `charts/all/observability/templates/` or values to add a GrafanaAlertRuleGroup:

* Query: histogram quantile over NeuroFace or OVMS request duration metrics
* Threshold: `> 0.5` seconds for 5 minutes
* Notification channel: your PagerDuty, Slack, or email receiver

. Ensure OpenTelemetry collectors export the required metrics from NeuroFace and gateway namespaces.

. On spokes, confirm `observability` chart scrapes inference ServiceMonitor endpoints.

=== Files affected

* `charts/all/observability/values.yaml`
* `charts/all/observability/templates/` (alert rule manifests)
* Optional: Thanos Ruler configuration for multi-cluster alert evaluation

=== Validation

[source,bash]
----
$ oc get grafanadashboards -n openshift-cluster-observability-operator
$ oc get prometheusrules -A | grep neuroface
$ curl -sk -u <grafana_user>:<password> \
  https://grafana.apps.<hub_domain>/api/v1/provisioning/alert-rules
----

.Expected output
[source,text]
----
neuroface-inference-latency-sla   openshift-cluster-observability-operator
[{"title":"NeuroFace inference p95 > 500ms",...}]
----

== Scaling workshop users

Workshop mode defaults to 30 users. Update `userCount` consistently in `platform-users`, `developer-hub`, `gitlab-operator`, `openshift-ai-hub`, `devspaces`, and `showroom` application overrides. See the pattern README for details.

== Optional Trusted Software Supply Chain

The pattern includes commented subscription stubs for RHTAS in `values-hub.yaml`. Enable them when your organization requires artifact signing as part of CI/CD.

For installation issues during customization, see xref:troubleshooting.adoc[Troubleshooting].
